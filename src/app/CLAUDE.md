## Organization Member Management

- Org owners can: list members, update member roles, remove members, invite new members.
- Org owners CANNOT: edit user data (name/email) or set passwords (admin-plugin-only operations).
- Member management UI on Account page (`/account`), visible only to org owners.
- Org-specific forms in `src/components/forms/form-set-org-role.tsx`, `form-remove-member.tsx`, `form-invite-member.tsx`.

## Platform-admin: user ↔ organization management (`/settings` → Users)

Platform admins (not org owners) manage *any* user's org memberships from the Users tab — distinct from the org-owner Account-page surface above.

- **⚠ Do NOT use the better-auth org client (`authClient.organization.updateMemberRole` / `removeMember` / etc.) for admin membership edits.** Those authorize against the **caller's** own membership in the target org, so a platform admin who isn't a member of that org fails with **"member not found"** (this was a real bug in the old "Org Role" dialog). Admin membership edits go through server functions that re-authorize with `requireAdmin()` and write the `member` table directly — same posture as `admin-organizations.ts` org edits.
- **Server** (`src/server/admin-organizations.ts`, all `requireAdmin()`-gated):
  - `getAdminUserOrganizations()` → `{ userOrgMap, userOrgDetails }` (per-user org list for the table).
  - `listOrganizationOptions()` → `{ id, name }[]` for the membership-editor picker.
  - `setMemberRole(memberId, role)` — one member's role (the quick "Org Role" dialog).
  - `setUserOrganizations(userId, desired[])` — **reconciles** a user's full membership set: inserts new, deletes dropped, updates changed roles in one pass (member ids minted with `randomUUID()`, `createdAt` set explicitly since the column has no default).
  - **Last-owner guard** (`assertNotLastOwner`): any removal/demotion that would leave an org with zero `owner` members throws a clear error — prevents an admin edit from orphaning an org.
- **API** (`src/app/api/admin/user-organizations/route.ts`): `GET` → maps; `GET ?orgOptions=1` → `{ organizations }`; `PUT { action: "setRole", memberId, role }` and `PUT { action: "setMemberships", userId, memberships[] }`. 403 on `Unauthorized`.
- **UI** (`src/components/tables/table-admin-users.tsx`): per-user **"Org Role"** quick dialog (`form-admin-set-org-role.tsx`, shown only when the user has ≥1 membership) + the always-available **"Organizations"** membership editor (`form-admin-edit-user-orgs.tsx`) that adds/removes orgs, sets a per-org role, and saves the whole config in one reconcile call. Both refresh the table + org map on success.

## Invitation Flow (end-to-end)

1. **Owner invites** via `InviteMemberDialog` (`src/components/forms/form-invite-member.tsx`) on the Account page. The form calls `authClient.organization.inviteMember({ email, role, organizationId, resend: true })`. The `resend: true` flag tells better-auth to refresh the existing pending invitation's `expiresAt` and re-send the email instead of returning 400 `USER_IS_ALREADY_INVITED_TO_THIS_ORGANIZATION` — so the admin can re-invite the same email any number of times without manual DB cleanup. The same `invitation.id` stays valid, so older email links still work.
2. **Better-auth fires `sendInvitationEmail`** (registered in `src/lib/auth.ts` on the `organization` plugin) → calls `sendEmails({ to, subject, body })` → routes through the Nylas provider by default, delivering from `hello@truffalo.ai`. The email body contains `${NEXT_PUBLIC_APP_URL}/accept-invitation/${data.id}`.
3. **Invitee opens the link** → `src/app/(auth)/accept-invitation/[id]/page.tsx` (client component). The page:
   - Fetches invitation details via the **public** `GET /api/invitations/[id]` endpoint (not better-auth's `getInvitation`, which is session-gated). The endpoint returns `{ id, email, organizationId, organizationName, status, expiresAt, expired }`.
   - Branches:
     - **Loading** → spinner.
     - **Not found / expired / status ≠ pending** → "Invalid invitation" card with Sign-in link.
     - **Not signed in** → inline **email-locked sign-up form** (`SignUpForInvitation`): Name + Password + Confirm. Email field is pre-filled with `invitation.email` and disabled so nobody can sign up under a different address. OTP and social login are intentionally not offered on this page (OTP is `disableSignUp: true`; social providers might return a non-matching email). Footer offers a Sign-in link with `callbackURL=/accept-invitation/<id>` for users who already have an account.
     - **Signed in, emails match** → "Accept invitation" button.
     - **Signed in, emails don't match** → "Wrong account" warning + **Sign out** button. After sign-out the page re-renders into the sign-up branch.
   - On successful submit of the sign-up form, runs in sequence: `authClient.signUp.email(...)` → `authClient.organization.acceptInvitation({ invitationId })` → `authClient.organization.setActive({ organizationId })` → push `/dashboard`. The `setActive` call is required because `session.create.before` picked `null` (no default org was created by the hook) — without it the dashboard would look empty.
4. **Pending-invite guard on standalone sign-up** — `src/components/forms/form-sign-up.tsx` calls `GET /api/invitations/check?email=…` on submit. If the endpoint responds `{ hasPending: true }`, the form shows a blocking amber banner ("You have a pending invitation. … please ask the organization administrator to resend it") and does **not** create the account. The banner auto-clears when the user edits the email field. If the check endpoint itself errors (network hiccup), the form **fails open** and lets sign-up proceed — the `user.create.after` hook still skips default-org creation, so the two-org outcome can't happen.
5. **Server functions** live in `src/server/invitations.ts`:
   - `getInvitationForAcceptance(id)` — public-safe lookup for the accept page; joins `organization.name` and exposes the minimal public fields plus an `expired` flag.
   - `hasPendingInvitationForEmail(email)` — boolean-only check, normalized to lowercase, filters to `status = "pending"` and `expiresAt > now`.

**Minor known edge case**: if an email has a pending invite but the user signs up via `/sign-up` while the check endpoint is unreachable, they'll end up with **zero** orgs (no default org due to the hook, and the invitation isn't auto-accepted). They can still recover by opening the invite link afterwards (they're already signed in with the matching email → Accept branch) or creating a new org manually. In normal conditions the banner blocks this case.

**Security trade-off on the `/api/invitations/check` endpoint**: it allows enumerating which emails have pending invites. This is a common pattern (GitHub/Slack do similar) and was accepted for UX. If you need to tighten it later: rate-limit, add a CAPTCHA, or do the check only server-side during sign-up and reject with the same message.

## Provider registry

Source providers are described in two paired modules. Adding a new integration requires touching exactly these three files (plus the per-provider handler implementation):

1. `ALTER TYPE source_provider ADD VALUE 'newthing'` migration in `src/db/schema.ts`
2. Metadata entry in `src/lib/sources/providers.ts` — `Record<SourceProvider, ProviderMetadata>` with `label`, `description`, `icon` (Lucide), `defaultName`, `defaultProviderConfig`, `defaultAutomatedParsingIsAllowed`, and the five `capabilities` flags below
3. Handler entry in `src/server/providers/handlers.ts` — `Record<SourceProvider, ProviderHandler>` with `sync` (or `null`), `isItemMissing` (or `null`), and `credentialsSchema` (Phase 3, currently `null` for all providers)

The `Record<SourceProvider, …>` typing on both registries makes step 2 + 3 compile-required as soon as step 1 lands — TypeScript fails the build until the new entries exist.

**Server vs client split:** `src/lib/sources/providers.ts` is intentionally client-safe (just metadata + Lucide icon refs). Client components import it directly. `src/server/providers/handlers.ts` is `import "server-only"` and pulls in the Nylas SDK / googleapis / drizzle — it must never be imported from a `"use client"` module.

**Capability flags — kept granular even where pairs currently move together:**

| Flag                          | What it gates                                                                                                                              | True today for                  |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------- |
| `supportsRemoteSync`          | `<SyncActionBar>` per-source Sync button + `runFullSync()` cron iteration filter (`syncableProviders()` in `src/lib/sources/providers.ts`) | nylas, gchat, gdrive            |
| `supportsArchiveUpload`       | The "Sync from WhatsApp Archive" button on the action bar (only renders if at least one seeded source matches)                             | whatsapp                        |
| `supportsDropoffUpload`       | The "Drop Off Your Files" dialog                                                                                                           | dropoff                         |
| `hasRawBytesPersisted`        | `<RowActions>` Re-parse availability — drop-off discards bytes after parse, aichat is in-memory                                            | nylas, gchat, gdrive, whatsapp  |
| `supportsAutomatedPipeline`   | Whether the daily cron sync/parse/upload pipeline iterates this provider. Distinct from `supportsRemoteSync` because an internal provider could in principle be auto-parsed without remote sync — kept separate so the distinction stays explicit. | nylas, gchat, gdrive, **telegram** |
| `supportsManualFetch`         | A per-source on-demand PULL button (`<TelegramFetchButton>` in `<SyncActionBar>`) distinct from `supportsRemoteSync` (which also drives the cron). Telegram is webhook-push in prod but offers a manual `getUpdates` pull, mainly for local dev. Kept OFF the cron. | **telegram**                    |

**Provider-specific call sites that bypass the dispatcher:** `src/server/parse-source-item.ts` calls `isNylasItemMissing` and `isGoogleChatItemMissing` directly inside the Nylas/Chat attachment-download helpers. Those branches are already inside provider-conditional code paths, so the matcher call doesn't need to go through `getHandler(ctx.provider).isItemMissing` — the dispatcher pattern only applies to the parent-row outer try/catch.

**Phase 2 (DONE):** `source_template` table holds the platform-wide dictionary; per-org `source` rows are instantiated from templates via `instantiateFromTemplate(orgId, templateId)` in `src/server/templates.ts`. `source.template_id` is a nullable FK back to the template (kept on `set null` so soft-deleting a template doesn't orphan instances). Admin manages templates at `/settings → Templates` (CRUD via `<FormAdminEditTemplate>`); org owners add sources via the new "Add source" picker on the manage tab (`<AddSourceDialog>` posts to `POST /api/sources/org/instantiate`). On new org creation, `databaseHooks.organization.create.after` calls `bootstrapDefaultsForOrg` to auto-instantiate every `is_default = true` template. Existing seeded sources were linked to templates via `scripts/backfill-template-ids.ts` (matched on `(provider, name)`). `getOrCreateAiChatSource` is now a thin wrapper around `instantiateFromTemplate` (idempotent). Owners can also rename their sources via `<FormSourceIdentity>` at `PATCH /api/sources/org/identity`. **Phase 3 (DONE):** every handler declares a `credentialsSchema` (zod) and the runtime path reads `source.credentials_ref` instead of `process.env`. Org owners + admins manage credentials via `<FormSourceCredentials>` (write-only, schema-driven) at `PUT /api/sources/org/credentials` and `PUT /api/admin/sources/credentials`. The one-shot migration `scripts/migrate-credentials-to-db.ts` seeds existing rows from env. nylas keeps an env fallback for the grant id; gchat/gdrive throw on null `credentials_ref`. See `src/lib/CLAUDE.md` § "Per-source credentials (Phase 3)" for the full env-fallback policy.

## Source templates (Phase 2 dictionary)

Templates are the platform-wide catalogue of source variants. The 7 seeded templates: Emails (nylas), Google Chat (gchat), Google Drive (gdrive), Google Meet (gdrive), Files Drop Off (dropoff), WhatsApp Archive (whatsapp), AI Chat (aichat). Multiple templates can share a `provider` — Google Drive vs Google Meet are both `gdrive` with different intent.

**Two policy flags** — kept separate so admins can decouple defaulting from visibility:

| Flag                  | Meaning                                                                                                       | True today for                  |
| --------------------- | ------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| `is_default`          | Bootstrap hook auto-instantiates this template into every newly-created org                                   | Files Drop Off, AI Chat         |
| `is_visible_to_orgs`  | Template appears in the org-owner "Add source" picker                                                          | All 7                           |

Phase 2 chose the **conservative defaulting** policy (Q5 (b)): only Files Drop Off + AI Chat auto-instantiate. Owners explicitly add Email / GChat / GDrive themselves via "Add source" — they have to provide their own credentials anyway, so forcing the explicit step matches the per-org tenant-isolation story Phase 3 set up.

**Default `provider_config` is empty `{}`** for every seeded template. Each org's own `driveId` / `spaceId` / etc. is filled in via `<FormSourceProviderConfig>` after instantiation. This keeps templates tenant-neutral — a new org pulling the Google Drive template doesn't inherit truffalo.ai's drive id.

**Soft-delete only**. Setting `status = 'inactive'` hides the template from the picker and from the bootstrap hook, but existing instances keep their `template_id`. Hard-delete cascades `template_id → null` on instances (FK on delete set null), but no UI exposes hard-delete.

**`instantiateFromTemplate(orgId, templateId, { idempotent? })`** copies `name`, `description`, `default_provider_config`, `default_automated_parsing_is_allowed`, `provider`, `type` from the template into a new `source` row scoped to the org. `credentials_ref` is always null at creation — credentials are per-org and set later. `idempotent: true` (used by the bootstrap hook + `getOrCreateAiChatSource`) short-circuits when a row with `(orgId, templateId)` already exists; default is non-idempotent (a second click in "Add source" creates a second instance, by design — same template can map to multiple drives/spaces in the same org).

**Bootstrap on org creation** lives in `databaseHooks.organization.create.after` in `src/lib/auth.ts`. Failures are caught + logged so a missing template seed never blocks org creation; the cost is the new org has no default rows until the seed script runs and the user creates a fresh org (or a server-side fixup is added). For day-to-day this means: production deploys MUST run the seed script before the new auth code lands.

**Adding a new template** = just an admin POST to `/api/admin/templates`. No code changes required (provider must already be in the registry). To add a new provider entirely, see `Provider registry` above.

## Sources scope, tabs & permissions

The Sources subsystem has two complementary management surfaces, gated by different roles:

| Surface                                         | Who                  | Where                                           | Edits                                                                |
| ----------------------------------------------- | -------------------- | ----------------------------------------------- | -------------------------------------------------------------------- |
| Platform admin sources dictionary               | platform `admin` role | `/settings` → Sources tab                      | full create / edit (provider, credentials, owner org, `isSystem`, `automatedParsingIsAllowed`, status) |
| Org-owner sources management                    | org `owner` role      | `/sources` → "Manage organization sources" tab | toggle `automatedParsingIsAllowed` and `status` on sources owned by the active org |
| Day-to-day operator UI (Sync / Parse / Upload)  | any authenticated org member | `/sources` → "Your organization sources" tab | per-row actions on items belonging to the active org                 |

**Source ownership model** (also summarised in root `CLAUDE.md` § Multi-Tenancy):
- `source.is_system = true` + `owner_organization_id = null` → platform-wide. Currently empty after the org-scoping refactor; the System tab on `/sources` keeps a coherent empty state for the future.
- `source.is_system = false` + `owner_organization_id = <orgId>` → per-org. All five seeded sources (Emails / Google Chat / Google Meet / Google Drive / Files Drop Off) live here, each cloned per organization that needs them. Drop-off lookups on `POST /api/sources/dropoff/upload` filter by `owner_organization_id = activeOrgId`.
- `source_item.organization_id` is denormalised from `source.owner_organization_id` at upsert + parse time so listing queries never need to join `source` for tenant scope.

**Trust boundary** (every server-side check):
- API routes call `getServerSession()` and read `session.session.activeOrganizationId`. Missing session / missing active org → 401.
- Per-row mutations and reads call `assertSourceInScope(sourceId, orgId)` (in `src/server/sources.ts`) or `assertSourceItemInScope(itemId, orgId)` (in `src/server/source-items.ts`). Both throw a typed `SourceScopeError` / `SourceItemScopeError` with `reason: 'not_found' | 'forbidden'` so the route layer can translate to 404 / 403 cleanly. System rows (org null) pass these checks too — currently unreachable for org members but kept callable for the future.
- Org-owner-only mutations call `requireOrgOwner()` (in `src/server/sources.ts`) which joins `member` to verify role; throws `OrgOwnerError` for unauthorised / non-owner.
- AI chat tools (`/api/chat`'s `searchSourceItems` / `getSourceItemContent` / `showSourceItemsInPanel`) take the active org id from the session and pass it through `listSourceItems({ organizationId })` + `getSourceItemMarkdown(id, { requireOrganizationId })`. Forged ids from another org return "not found" rather than leaking content.

## Platform admin sources dictionary (`/settings`)

- **Page**: `/settings` → Sources tab (platform-admin only). The form/route handle the full source schema; this is the only surface that creates new sources or edits credentials.
- **DB**: `source` table holds the dictionary; `source_item` holds individual fetched items (see `src/db/schema.ts`).
- **Server**: `src/server/admin-sources.ts` — `listAdminSources`, `createAdminSource`, `updateAdminSource`, `listOrgOptionsForAdmin`. All gated by `requireAdmin()`.
- **API**: `src/app/api/admin/sources` — GET (list with filters / pagination, or `?orgOptions=1` for the form's org selector), POST (create), PUT (update by `sourceId`).
- **Table**: `src/components/tables/table-admin-sources.tsx`. Columns: Type / Provider / Organization / Name / System / Status / **Auto Parse** / Actions. Filters: name search, provider select (sentinel `__all__`), `Show inactive` checkbox (omitted by default — soft-deleted sources hidden). Pagination 5/page. "Add new" button at the right of the filter row.
- **Form**: `src/components/forms/form-admin-edit-source.tsx`. Reused for create + edit (no `source` prop = new). Includes the **Auto Parse** checkbox alongside System source / status. The provider-config field stays a generic JSON textarea (it holds connection identifiers like `spaceId` / `driveId`); credentials moved to a dedicated schema-driven dialog (`<FormSourceCredentials>`) — same component the org-owner table uses.
- **Credentials handling — write-only, schema-driven**:
  - The decrypted credentials are NEVER sent to the browser. The list endpoint returns `hasCredentials: boolean` only.
  - On edit, a "Configure / Replace" button next to the Provider field opens `<FormSourceCredentials>` which renders per-provider inputs (Nylas: grantId; gchat: serviceAccountJson + impersonateUser; gdrive: serviceAccountJson). The dialog posts to `PUT /api/admin/sources/credentials` which validates against the per-provider zod schema in `src/server/providers/handlers.ts`, encrypts via `credentials-crypto.ts`, and writes to `credentials_ref`. The dialog is hidden in create mode (no source row id yet) — the admin saves the row first, then opens it in edit mode to configure credentials.
  - Runtime parsers + sync handlers now read from `credentials_ref` (Phase 3). Adding a row through this form WITH credentials configured grants the parsers access immediately; rows without credentials throw `MissingCredentialsError` at sync/parse time for gchat/gdrive (Nylas falls back to `NYLAS_GRANT_ID` env with a warning).
- **Soft delete**: setting `status = inactive` hides the source from the table (unless the admin ticks "Show inactive"). No hard-delete UI — cascading would also drop `source_item` rows + R2 markdown.
- **Encryption**: `credentials_ref` is AES-256-GCM ciphertext (base64) via `src/lib/credentials-crypto.ts`. Wire format carries a 1-byte version prefix (v1 = `0x01`) for future rotation. Key from `CREDENTIALS_ENCRYPTION_KEY` env var.

## Org-owner sources management (`/sources` → "Manage organization sources" tab)

- **Visibility**: tab is rendered only when the caller's role in their active org is `owner` (resolved server-side via `getActiveOrgRole()` and threaded into `<SourcesPageShell isOrgOwner>`). Members + admins-without-owner-role don't see the tab. The trust boundary is the API route, not the prop — the underlying `/api/sources/org` route does its own `requireOrgOwner()` check.
- **Server**: `src/server/sources.ts` exposes `listOwnerOrgSources()` (returns identity + editable knobs only — no credentials, no provider_config) and `updateOwnerOrgSource(sourceId, patch)`. Both go through `requireOrgOwner()` and verify the target row's `ownerOrganizationId` matches the caller's active org before mutating.
- **API**: `src/app/api/sources/org/route.ts` — GET (list owner-editable rows for the active org) + PATCH (`{ sourceId, automatedParsingIsAllowed?, status? }`). Errors are translated from `OrgOwnerError` → 401/403 and `SourceScopeError` → 404/403.
- **Table**: `src/components/tables/table-org-sources.tsx`. Columns: Provider / Name / Status / Auto Parse / Last Synced. Status + Auto Parse are inline `Switch` toggles that PATCH on change with optimistic local update + server-truth re-fetch on failure. The switch palette is overridden by the local `NEUTRAL_SWITCH` className (white-ish rail + dark thumb on checked) so the orange theme accent lives entirely on the adjacent `Badge` rather than the rail itself.
- **What this tab is intentionally NOT**: it doesn't expose credentials, provider config, owner org, or the `isSystem` flag. Those edits stay on the platform-admin Settings page; the owner-side knob set is deliberately limited to schedule + soft-disable.

## Sources page layout

- **`src/app/(protected)/sources/page.tsx`** is a **server component** that loads everything the shell needs in one parallel block: the active org id from `getServerSession()`, the system source list (`listSystemSources()`), the org source list (`listOrgSources(orgId)`), and the active-org role (`getActiveOrgRole()`). Hands all of it to `<SourcesPageShell>`. The admin-only `<WorkflowStatistics>` card renders below the shell, gated on `session.user.role === 'admin'` so non-admins don't see the inevitable 403 from the admin-gated stats endpoint. Container is `max-w-7xl px-4`.
- **`src/components/blocks/sources-page-shell.tsx`** is the client wrapper that owns cross-component state (`refreshKey` bumped on each Sync / per-row action / dropoff upload; `uploadOpen` for the controlled dropoff dialog). The page is split across three shadcn `Tabs`:
  1. **"Источники организации" (Your organization sources)** — default tab. Renders `<SourcesScope>`: action bar (`<SyncActionBar>`) → shared `<ProcessRunBar>` → **one unified `<TableSourceItems>` card** («Материалы источников»). **Unified parse+upload model (2026-06):** the old separate Pending/Processed cards + «Разобрать все»/«Загрузить все» controls were collapsed into a single table that shows BOTH sub-process statuses (a **Разбор** badge + a **Загрузка** badge per row) and a single smart **«Обработать»** action that does whatever remains (parse → upload, or just upload). Sync buttons became **«Синхронизировать и обработать»**: they POST `/api/sources/sync`, then chain straight into the parse→upload run scoped to that source. A **«Период обработки»** date-range control (from/to, `YYYY-MM-DD`, default empty = all) sits in the action bar next to the sync buttons and bounds processing by `source_created_at` (threaded into `useProcessRun.run({ dateFromIso, dateToIso })` → `/api/sources/items/process-ids` → `listProcessItemIds`'s `dateFrom`/`dateTo` `sourceCreatedAt` filter). State lives in `<SourcesScope>` and is the single source of truth for the date bound across **all three** processing triggers: the per-source sync→process chain (`onProcessSource` closure), the table's «Обработать все», AND the `<TableSourceItems>` listing itself (passed as `periodFrom`/`periodTo` props → effective date bound applied on EVERY view, superseding the table's own date inputs — which are hidden behind a read-only chip while a period is set). So selecting a period scopes what you SEE and what you PROCESS to the same window. Empty period → the table reverts to its own per-view date inputs. See § "Unified source processing" below for the mechanics.
  2. **"System sources"** — `<SourcesScope scope="system">` renders the same shape against the (currently empty) `is_system = true` bucket. Kept around for future platform-wide sources; shows a "No system sources configured" empty card today.
  3. **"Manage organization sources"** — owner-only (gated by `isOrgOwner` from `getActiveOrgRole()`). Renders `<TableOrgSources>` — see § "Org-owner sources management" above.
- **`<SourcesScope>`** (private to the shell) — renders empty-state when `sources` is empty, otherwise mounts the action bar + `<ProcessRunBar>` + one `<TableSourceItems>` card. It owns a single `useProcessRun({ onRefresh })` instance shared by the per-source sync buttons AND the table's "Обработать все" control, so only one parse→upload run is ever in flight and both feed the same progress bar.
- **`<TableSourceItems>`** (`src/components/tables/table-source-items.tsx`) is now the **single unified table** (the old `status: pending|processed` / `scope` split is gone — it's always the active org). State: a **`view` filter** (`Требуют обработки` [default] / `Готово` / `Ошибки` / `Все` → API `view=needs_work|done|errors|all`), source filter, free-text search, pagination (10/page). Date inputs appear only for the unbounded views (`done`/`all`, defaulted to today); `needs_work`/`errors` drain to zero so they're shown without a date bound. Two per-row badges (`ParseBadge` + `UploadBadge`) show both sub-processes; a smart **«Обработать»** button calls `POST /api/sources/items/[id]/process`, plus Show (when parsed) + Re-parse (roots with persisted bytes). Per-row actions are disabled while a shared batch run is in flight.
- **`/api/sources/items`** (GET): send EITHER `view=needs_work|done|errors|all` (unified table) OR the legacy `status=pending|processed`. Plus `scope=system` (omitted = active org), `sourceId`, `q`, `date_from`, `date_to`, `limit`, `offset`. Returns `{ rows, total }`. `listSourceItems` (in `src/server/source-items.ts`) requires `organizationId` (string / `null` system) so cross-tenant listing is impossible. The `view` clause derives from the two status columns (see `viewClause`); `needs_work` = parse pending/processing/failed OR parsed-but-unshipped, `errors` = parse failed OR upload failed, `done` = complete+uploaded OR skipped. Route auth: 401 if no session / no active org.
- **`/api/sources/sync`** (POST): body `{ sourceId }`. Verifies the source belongs to the caller's active org via `assertSourceInScope`, then dispatches by provider via `syncSource()` (`src/server/sync/index.ts`). `dropoff` rejects since user-uploaded files have no provider to fetch from. Per-provider sync functions in `src/server/sync/{nylas,gchat,gdrive}.ts` each apply the `max(sourceCreatedAt) - 5min overlap` cursor strategy and call `upsertSourceItem` (which touches metadata fields only — never resets `parseStatus` / `r2UploadStatus`).

## Own-org attribution of source items (parse-time)

Every source item is classified at parse time as authored **by the owning org (`own_org`)**, by an **outside party (`external`)**, or **`unknown`** — powering first-party / internal-document filtering (e.g. future AI-chat "company knowledge base" scoping). Spec: `refs/org-attribution.md`.

- **The one rule — directionality**: attribution answers *"who AUTHORED this?"*, never who is mentioned/who received it. Only the **sender/author** side counts, so an inbound client email addressed to the team reads `external`, not `own_org`. The per-provider `authorEmailField` (in `src/lib/sources/providers.ts`) MUST point at the sender field only: `from` for nylas/imap (envelope sender), `participants` for gchat/gdrive (message sender / file owner), `null` for dropoff/whatsapp/aichat/telegram (no author side → classified by content).
- **Identity** (`src/server/org-identity.ts` → `getOrgIdentity`, memoized ~5 min, `invalidateOrgIdentity` to drop): defines "us" = human-member emails (agent accounts `user.role==='agent'` excluded) + the non-freemail domains the org owns (member domains + `org.web_url` host + `org.email` domain, every one freemail-guarded so a member's `@gmail.com` can't poison the set). Shares the `organization` source with the discovery-side `loadOwnOrgIdentity` (different consumer).
- **Classifier** (`src/server/parsers/org-attribution.ts`, `import "server-only"` — NOT `"use server"`, exports sync fns + a type): `extractAuthorEmails(metadata_json, provider)` (registry-driven) → `classifyOrgAttribution` (strict priority: ① author ∈ member emails → own_org/high; ② author's non-freemail domain ∈ owned domains → own_org/high; ③ there IS an author, neither → external/high; ④ **no** author + org website host appears in content → own_org/**medium**; ⑤ else unknown/low) → `resolveOrgAttribution` orchestrator: blocklist override (a blocklisted author forces `external`/high unless already own_org), **deterministic `high` short-circuits before the LLM** (email/chat with a resolvable sender never pays), then the **LLM authorship judge** lifts only the ambiguous no-author document tail (`Output.object`, content ≤8k; its crux is *mentioning the org ≠ authorship*; abstains to `unknown` on error so the deterministic verdict stands; can upgrade an `unknown` doc OR correct a weak `org_url` own_org that's actually third-party). `matchedOn` tags (`sender_member_email` / `sender_org_domain` / `external_sender` / `org_url` / `blocklist_author` / `llm_judge`) are stable + searchable.
- **Pipeline** (`src/server/parse-source-item.ts`): `computeOrgAttribution(ctx, markdown)` runs for parents (bails `unknown` with no org; `enableLlmJudge: provider !== 'whatsapp'`). `markParsed` writes the `org_attribution` column AND folds the full `{value,confidence,matchedOn,reason}` evidence into `metadata_json.orgAttribution` (right-biased jsonb merge). **Children inherit** via `ctx.orgAttributionValue` (set during the parent's markParsed, which always runs before children are inserted — an attachment of our own email is ours; covers the video→derived-audio grandchild too). `reparseSourceItem` resets `org_attribution='unknown'` so the next parse recomputes.
- **Schema**: `org_attribution` enum + `source_item.org_attribution` column (NOT NULL default `'unknown'`, so existing rows fill on migration with no backfill) + a partial index `source_item_org_attribution_own_idx ON (organization_id) WHERE org_attribution='own_org'` keeping first-party lookups tiny. Applied additively via direct SQL.
- **Note**: the spec's `recordLlmUsage` telemetry hook doesn't exist in this codebase, so the judge runs without usage recording (no behavioural difference; add the hook later if per-org LLM cost tracking lands).

## Source teardown / reset (admin-only)

Admin tool to hard-delete selected **threads** of a source — a thread = one top-level `source_item` (e.g. a test email) **+ its children** — together with the cards + clients/contacts/deals/tasks/orders those threads EXCLUSIVELY produced, so the **same test/demo scenario re-runs cleanly** with no dedup/attribution collisions. Tab «Сброс источника» on `/settings` (platform-admin only). Spec: `refs/source-teardown.md`. **The unit is the thread, not the whole source** — selecting threads deletes their items AND their entities as one bundle (no orphaning), and full-source reset = "select all threads".

- **Why hard delete + re-aggregation**: soft-`delete` is *revived on rescan* (the exact collision being fought) and `blocked` suppresses rediscovery — both defeat "as if new". And only **cards** store a source link (`card.source_item_id`); clients/contacts/deals have **no provenance column**. So the deletable set is reconstructed by **re-aggregating the discovery collision keys** from `metadata_json` (no LLM): for each existing client/contact, `producing` = the org-wide set of `source_item`s whose `companyMatchKey` / email / `personMatchKey` it matches — exactly the set the next run would dedup against. **Soft-deleted (`deleted`/`blocked`) clients/contacts ARE included** in the match (a reset clears tombstones too). Manually-created rows match no source item → untouched for free.
- **Shared-safe deletion (the load-bearing rule)**: an entity is deleted **iff every item in its `producing` set is inside the selected threads** (i.e. `producing ⊆ selectedItemIds`). If any producer lies in an unselected thread OR another source, it's **kept** and counted as `sharedSkipped` — you can never corrupt a scenario you're not deleting. Server-authoritative: `executeSourceTeardown` re-resolves and re-decides; a stale/forged `threadIds[]` can't widen the set.
- **Server** (`src/server/teardown.ts`, `import "server-only"` — **not** `"use server"`, since it exports a class + sync helpers): `requireAdmin` (`user.role==='admin'` → typed `TeardownError` → 401/403/404/400); a **self-contained** metadata reader (`companyKeysFromMeta` / `contactKeysFromMeta` reimplement discovery's extraction); `resolve(sourceId)` groups the source's items into threads (parent + `parent_source_item_id` children), builds the org-wide key→items maps, and computes each client/contact's `producing` set (clients also `orderCount`). `previewSourceTeardown` returns a `threads[]` tree, each thread carrying its `clients`/`contacts` with provenance badges (`exclusiveToThread`, `otherThreadsInSource`, `itemsInOtherSources`, `producingThreadIds` — the last lets the UI compute the exact live delete-estimate). `executeSourceTeardown({sourceId,confirmText,threadIds})` re-validates `confirmText === source.name`, intersects requested threads with resolved, deletes the shared-safe set.
- **FK-safe delete order** (load-bearing — `deal.client_id`/`order.client_id`/`order_request.client_id` are FK-`restrict`, `card.source_item_id` is `set null`, `contact.client_id`/`task.*` are `set null`): R2 objects (`deleteFromR2`, selected threads' keys) → tasks → deals (cascades `deal_contact`) → **`order` (cascades `order_item` + `order_access_link`) + `order_request` (cascades `order_request_item`)** for the deleted clients → cards whose `source_item_id` ∈ selected items (cascades `card_*` junctions) → contacts → clients → the **selected** `source_item`s (by id; children included in the set) → reset `source.last_synced_at = null` **only on a full-source select** (partial leaves the cursor so kept threads aren't re-pulled) → insert `teardown_log`. The source row is **kept**.
- **Schema**: `teardown_log` (append-only audit: org, source id+name, admin user, `counts jsonb` — now a `TeardownResult` with `threads`/`sharedSkipped`/`cursorReset`, never read by app logic) + `deleteFromR2(keys[])` in `src/lib/r2.ts`. Applied additively via direct SQL.
- **API** (`/api/admin/teardown/*`, admin-gated): `GET …/sources` → cross-org picker `{id,name,provider,organizationId,organizationName,itemCount}`; `POST …/preview {sourceId}` → `{ source, threads[] }`; `POST …/execute {sourceId,confirmText,threadIds[]}` (`maxDuration=300`) → the `TeardownResult` counts.
- **UI** (`src/components/blocks/admin-teardown.tsx`, Russian): source `Select` → preview renders a **grouped thread tree**. Each thread is a checkbox row (title + date + item/card chips); under it its clients/contacts with badges «только этот тред» / «⚠ ещё в N тред.» / «⚠ в др. источнике — не удалить» (the last never deletable here), a rose «заказов: N» chip, an «уже удалён» chip. A **«Выбрать все треды»** master toggle + a live, server-matching estimate (Элементы / Карточки / Клиенты / Контакты / ⚠ оставлено общих). A **typed-confirmation** input arms the destructive button. Execute → toast deleted + shared-skipped counts → reload picker + re-preview.
- **Caveat**: re-aggregation is a *reconstruction*, not recorded lineage — a client renamed after creation could key differently; the preview→confirm flow is the safety net. Resolution is O(org size) (scans the org's `metadata_json`) — fine at demo/pilot scale. For exact future teardowns, an `entity_source_item` lineage junction (written at creation) would be the upgrade; re-aggregation stays primary for existing data.

## Source Parsing (admin-only)

- **Goal**: convert fetched source items (emails, chats, drive files, dropped files) into a structured markdown document matching `refs/parsing-sources-template.md` (YAML frontmatter + `## Summary` + `## Content`). Used manually today; will be called by schedulers/AI agents once automation is wired up.
- **Config**: `src/lib/parser-config.ts` holds model IDs and tuning constants (e.g. `PARSER_CONFIG.pdf.maxBytes = 20 MB`). Will migrate to a DB-backed settings dictionary once the schema is ready.
- **Shared helpers**: `src/server/parsers/_shared.ts` exports `buildFrontmatter`, `assembleMarkdown`, `uniqueStrings`, `extractUrls`, `emailsToDomainUrls`, and the `SourceFrontmatter` type — every parser produces frontmatter via this module so the template format stays consistent across source kinds.
- **Source ID namespaces** (all emulated until the DB table lands — the namespace prefix keeps them visually distinct from future real UUIDs):
  - `nylas:<messageId>` — email body
  - `nylas-att:<messageId>:<attachmentId>` — email attachment (including CID-referenced inline images — Nylas surfaces those as real attachments with `isInline: true`)
  - `nylas-att:<messageId>:<attachmentId>:audio` — audio track extracted from a video attachment (its `parent_source_id` points to the video block itself, not the email)
  - `nylas-inline:<messageId>:<index>` — image embedded directly in the body HTML as a `data:image/…;base64,…` URI (never a Nylas attachment, so it has no attachment ID)
  - `gchat:<messageId>` — Google Chat message body (messageId is the last segment of `spaces/X/messages/Y`)
  - `gchat-att:<messageId>:<index>` — Google Chat attachment (index within the message, since Chat attachment resourceNames are long opaque strings)
  - `gchat-att:<messageId>:<index>:audio` — audio track extracted from a Chat video attachment
  - `drive:<fileId>` — Google Drive file (body block; used for every Drive format — Docs, Slides, Sheets, and uploaded non-Google files like PDFs/images/videos/etc.)
  - `drive:<fileId>:audio` — audio track extracted from a video file stored on Drive
  - `dropoff:<entryId>` — file uploaded via the drop-off dialog (entryId = server-generated UUID minted in the upload route, persisted as `source_item.external_id`)
  - `dropoff:<entryId>:audio` — audio track extracted from a dropped video file

### Drop-off parser (user uploads)

- `src/server/parsers/dropoff.ts` — `parseDropoffFile({ bytes, fileName, mediaType, dropoffId, … })` dispatches to the neutral media parsers (PDF, image, audio, video, Word, PowerPoint). No Google-native handling — drop-off files are local uploads, not Workspace resources. Returns `{ blocks, fileName, mediaType, byteSize, dropoffId }` with 1 block for most formats or 2 for videos (summary + transcript). Source system = `"Dropped File"`.
- **Inline upload + parse**: `POST /api/sources/dropoff/upload` (multipart, org-scoped, `maxDuration = 300`) reads the form's `files[]` + `description`, looks up the dropoff source for the caller's active org, runs `parseDropoffFile` per file, and inserts a root `source_item` row (kind `dropoff_file`) with `parseStatus = 'complete'` + `parsed_markdown` for the body block. Video files emit a second block which is inserted as a child row with `externalType = 'derived_audio'`. The row lands directly in the Processed table — Drop-off never sees Pending. The route returns `{ results, okCount, failedCount }` so the dialog can toast per-file outcomes.
- **No raw-byte persistence**: bytes are discarded after the upload-time parse. Re-parse is therefore unsupported on `dropoff_file` rows — `<RowActions>` hides the Re-parse button when `row.sourceProvider === 'dropoff'`. The user re-uploads instead. `parseSourceItem(itemId)` still throws for `dropoff` items, since the row already arrived parsed and there's nothing for the generic parse pipeline to do.

### Drive parser (Google Workspace + any uploaded file)

- `src/server/parsers/drive.ts` — `parseDriveFile(fileId)` returns `{ blocks, fileName, mimeType, fileId }`. Drive files are standalone sources (no parent message, no thread), so every block carries `parent_source_id: null` and `thread_id: null`.
- **Dispatch by mime**:
  - **Google Docs** (`application/vnd.google-apps.document`) → `drive.files.export({ mimeType: "text/markdown" })` — Google added native markdown export in 2024, so content conversion is free. We only run **one Gemini call for metadata-only fields** (summary/mentions/companies/products/language) via a slim `metadataOnlySchema` — no content-conversion LLM pass, much cheaper than PDF/image/audio flows.
  - **Google Slides** (`application/vnd.google-apps.presentation`) → export as `.pptx`, delegate to `parseOfficeBytes` (pptx path). Zero new code.
  - **Google Sheets** (`application/vnd.google-apps.spreadsheet`) → export as `.csv`, parse with a tiny custom CSV parser (handles quoted fields with commas/escaped quotes), render as a **markdown table of column headers + first 10 data rows**, append total row count. `SHEETS_PREVIEW_ROWS` constant controls the preview depth. Then the usual metadata-only Gemini call.
  - **Uploaded files** (PDF / JPEG / PNG / MP3 / MP4 / .docx / .pptx on Drive) → `files.get({ alt: "media" })` to download raw bytes, then route through the matching existing parser (`parsePdfBytes`, `parseImageBytes`, `parseAudioBytes`, `parseVideoBytes`, `parseOfficeBytes`). No duplication.
  - **Folders / Forms / Shortcuts / Drawings** → throw `UnsupportedDriveTypeError`, route converts to `{ skipped: "unsupported type (…)" }` response.
- **Deterministic metadata** comes from Drive API: `senders = owners[].displayName/email`, `sourceCreatedAt = createdTime`, `urls` seeded with `webViewLink` plus any regex-extracted URLs from the content. `recipients` stays empty (Drive files don't have an "addressee" concept; permissions list could populate it later if useful).
- **Blocks array**: most formats return 1 block; videos return 2 (visual + transcript) matching the video parser's dual-block output.
- The Drive parser is invoked via `parseSourceItem(itemId)` (`src/server/parse-source-item.ts`) — there's no separate per-provider HTTP route anymore; the unified `POST /api/sources/items/[id]/parse` endpoint dispatches by `source.provider`.

### Chat parser (Google Chat messages)

- `src/server/parsers/chat.ts` — `parseChatMessage(messageName)` fetches a single message via `chat.spaces.messages.get({ name: "spaces/X/messages/Y" })`, extracts deterministic fields (sender, thread, dates, URLs), and calls Gemini 2.5 Flash for the content-derived fields. Mirrors the email text parser shape — same schema minus HTML conversion since Chat messages are plain text.
- **`senders` = `[message.sender.displayName]`** (single-author messages). **`recipients` = channel member displayNames via best-effort `listChatMembers(spaceName)`** — exclude bots + the sender; fall back to `[]` on any failure (missing scope, transient error).
- **Channel members**: requires `https://www.googleapis.com/auth/chat.memberships.readonly` authorised in the service account's DWD scope list (alongside the existing `chat.bot` + `chat.app.messages.readonly`). The scope is added to `APP_SCOPES` in `src/lib/google-chat.ts`; if the Workspace admin hasn't authorised it yet, `listChatMembers` logs a warning and returns `[]`, which is valid per the template.
- Returns `{ markdown, metadata, attachments, messageName, messageId, spaceName, threadId, sourceCreatedAt }`. The route chains attachments through the same dispatcher the email route uses.

### Chat attachment bytes

- `src/lib/google-chat.ts` — `downloadChatAttachmentBytes(resourceName)` is the shared helper both `/api/chats/attachments` (browser download proxy) and the chat parser's attachment dispatcher use. Impersonates `GOOGLE_CHAT_IMPERSONATE_USER` via DWD with `chat.messages.readonly`, hits `https://chat.googleapis.com/v1/media/<resourceName>?alt=media` directly (googleapis SDK's wrapper doesn't reliably include `alt=media`). Returns `Uint8Array<ArrayBuffer>` so the bytes are directly usable as `NextResponse` body and as parser input.

### Text parser (email bodies)

- `src/server/parsers/text.ts` — `parseEmailMessage(emailId)` fetches the full Nylas message, extracts deterministic fields (thread_id, dates, senders, recipients, URLs) directly from the payload, and calls Gemini 2.5 Flash (`google/gemini-2.5-flash` via Vercel AI Gateway) with `generateText` + `Output.object({ schema })` to produce the LLM-derived fields (`summary`, `mentions`, `companies`, `products`, `language`, clean markdown body).
- **Why hybrid extraction**: Nylas already gives us exact `from`/`to`/`cc`/`bcc`/`date`/`threadId` — routing those through the LLM would cost tokens and risk hallucination. LLM only handles content-derived fields.
- **URLs field** is the union of URLs extracted from the body HTML **and** `https://<domain>` for every sender/recipient email domain (deduplicated).
- **Image handling — three sources, one pipeline**: emails embed images in three ways, and all three are surfaced so the route can feed each to `parseImageBytes`:
  1. **Regular image attachment** (user attached a screenshot) — normal Nylas attachment, `isInline: false`.
  2. **CID-referenced inline image** (`<img src="cid:…">`) — still a Nylas attachment, just flagged `isInline: true`. We intentionally keep these (the previous `!a.isInline` filter was too aggressive — it hid signature photos and screenshots embedded via CID). Both flow through `attachments[]` via the normal Nylas `downloadBytes` path.
  3. **Data-URI image** (`<img src="data:image/png;base64,…">`) — not a Nylas attachment at all, encoded directly in the HTML. Scanned out of `msg.body` via `extractBodyDataUriImages()` (regex for `data:image/(jpeg|jpg|png);base64,…`, dedup by base64 payload, decoded to `Uint8Array` server-side). Returned separately as `bodyInlineImages: InlineBodyImage[]` so the caller can parse them without any Nylas round-trip.
- Returns `{ markdown, metadata, attachments, bodyInlineImages, messageId, threadId, sourceCreatedAt }`. The route chains both lists into parser dispatch.
- **Decorative inline-image triage** (`src/lib/image-triage.ts` + `PARSER_CONFIG.image.decorative`): emails are full of chrome — company logos, signature graphics, social badges, header/footer banners, divider rules, 1×1 tracking pixels — that each used to become a low-value `inline_image`/`attachment` source item (and an LLM bill). Two filters drop them, applied **only to inline images** (Nylas cid attachments where `isInline===true`, Nylas/IMAP data-URI `bodyInlineImages`); **explicit file attachments bypass both** (intentional content). (1) **Pre-LLM, zero-cost**: `triageInlineImage` reads pixel dimensions straight from the header bytes (`readImageSize` — PNG/JPEG/GIF/WebP/BMP, no dependency) and skips tracking pixels (≤3px), logos/icons (≤200px both axes), and thin banners (short side ≤120 & aspect ≥4.5), plus a conservative decorative-filename fallback. Unreadable headers → KEEP (defer to the LLM). (2) **LLM net**: the image schema now carries `isBoilerplate` — a high-res logo/banner that slips past the size gate is judged decorative and skipped. Both paths record an **audit row** via `insertSkippedChild` (reason on `parse_error`), so the image stays visible in Stored Content but is excluded from parsed content + discovery. Wired in `parse-source-item.ts` via the shared `parseInlineImageOrSkip` (data-URI sites) and the `isInlineImage` branch of `runAttachmentParser` (Nylas cid attachments).
- **IMAP email (`imap` provider) — second email source, no Nylas.** Same registry pattern as every other provider (enum + `PROVIDERS.imap` metadata + `HANDLERS.imap`). Per-org credentials `{ host, port, secure, user, password }` in `credentials_ref` (zod `imapCredentialsSchema`; **no env fallback** — strictly per-org, throws `MissingCredentialsError` if unset, like gchat/gdrive); the mailbox folder is `provider_config.mailbox` (`imapProviderConfigSchema`, default `INBOX`). Client: `src/lib/imap.ts` `buildImapClient` over **imapflow** (caller owns connect/logout). **Sync** (`src/server/sync/imap.ts` `syncImapEmails`) is incremental + **envelope-only** — `external_id = "<uidValidity>:<uid>"` (UIDVALIDITY is the load-bearing correctness invariant: UIDs are only stable within a mailbox generation), cursor = newest `source_created_at − 5 min`, `metadata_json` carries `{ mailbox, uid, uidValidity, messageId, subject, from[], to[], cc[] }` in the same `{name,email}` shape discovery's `extractParticipants` reads. The body is **not** stored — fetched lazily at parse. **Parse** (`src/server/parsers/imap.ts` `parseImapMessage`) reconnects, **re-verifies UIDVALIDITY** (mismatch / deleted uid → `ImapMessageMissingError` → row `skipped`), `fetchOne(source:true)` → `mailparser.simpleParser`, then runs the **shared** email LLM pipeline `analyzeAndAssembleEmail` (exported from `text.ts`) so IMAP and Nylas produce identical markdown + `MetadataAnalysis`. Attachment bytes are in-hand (no per-file download); `.ics` invites fold into the parent (see "Calendar (.ics) invites" above); `related`/cid inline images come through the data-URI path (mailparser inlines them into the HTML). `isImapItemMissing` handles the `NONEXISTENT` / "no such message" cases. Seed: `scripts/seed-imap-source.ts` (+ a row in `seed-templates.ts`). Spec: `refs/imap-source.md`.
- **Calendar (.ics) invites — folded into the parent email, not a child** (`src/server/parsers/ics.ts`, via `node-ical`): when the email carries an iCalendar attachment, `extractCalendarEvent` downloads the first `.ics` (one extra `nylas.attachments.downloadBytes` call) and decodes the first VEVENT. The invite enriches the SAME email — no new table, no child row, no schema change: (1) **roster** → `metadata_json.participants: [{email,name}]` from the ICS organizer + attendees (deduped, `ROOM`/`RESOURCE`/`GROUP` CUTYPEs dropped) — discovery's `extractParticipants` already reads that canonical shape, so attendees become contacts/clients with no discovery change; (2) **topic** → a deterministic `## Meeting` section is prepended to the body markdown (`mergeMeetingIntoContent`) and an invite block is injected into the LLM prompt (`buildCalendarContext`) so org/people extraction works even with no human body; (3) **junk override** — a real VEVENT forces `relevance.isJunk = false`, so a "bare" invite (no body, otherwise classified `calendar_only`) is parsed instead of skipped. The `.ics` itself is filtered out of `attachments[]` (already folded in; the format dispatch has no `.ics` handler). `ics.ts` is pure / `server-only` / never-throws — a malformed invite or failed download just yields `null` and the email parses normally. Spec: `refs/calendar-invites.md`.

### Office parser (universal, .docx + .pptx)

- `src/server/parsers/office.ts` — `parseOfficeBytes({ bytes, fileName, mediaType, … })` handles both Word and PowerPoint formats through one unified parser with format-specific extraction.
- **Two-stage pipeline** (no gateway model accepts Office formats natively — they're ZIP archives with XML inside):
  1. Server-side extraction — `mammoth` (BSD-2-Clause) turns `.docx` into HTML preserving heading hierarchy, lists, tables, bold/italic. For `.pptx`, a custom ~30-line extractor using `jszip` + `@xmldom/xmldom` (both MIT) walks the ZIP, sorts `ppt/slides/slideN.xml` files by slide number, and collects `<a:t>` text nodes. Slide boundaries are preserved as `=== Slide N ===` markers so the LLM can emit `### Slide N` sub-headings.
  2. Gemini 2.5 Flash cleanup + metadata — same schema and `Output.object` call as the PDF parser, just with pre-extracted content instead of a native file part.
- Content passed to Gemini is capped at 120k chars (~30k tokens) to guard against pathological docs; excess is clipped with a truncation marker.
- **Two UI kinds** from one parser: `docx` → "Word document" + `BookOpen` icon; `pptx` → "Presentation" + `Presentation` icon. Detection via `detectOfficeFormat()` by mime or extension.
- Legacy `.doc` / `.ppt` (pre-2007 binary OLE) are intentionally unsupported — they require a full Office rendering engine and are rare today.
- **Ambient declarations**: `src/types/mammoth.d.ts` provides minimal types for mammoth (no `@types/mammoth` exists on DefinitelyTyped).
- Size cap from `PARSER_CONFIG.office.maxBytes` (20 MB). Oversize → `OfficeTooLargeError`. Wrong type → `UnsupportedOfficeTypeError`. Both are converted to `skipped` entries by the route.

### Video parser (universal, MP4)

- `src/server/parsers/video.ts` — `parseVideoBytes({ bytes, fileName, mediaType, videoSourceId, audioSourceId, … })` mirrors the other parsers but **produces TWO markdown documents from a single Gemini call**: a visual summary block and an audio-transcript block. Gemini 2.5 Flash handles video natively — no ffmpeg audio-extraction step, no two-stage pipeline.
- **Single schema, two rendered blocks**: the model returns `videoSummary` + `videoContent` (setting, actors, key moments, emotional tone as Markdown) alongside `audioSummary` + `segments` + `speakers` (same structure as the audio parser). The parser assembles both into template-shaped markdown and returns `{ videoMarkdown, audioMarkdown, metadata }`.
- **Parent link**: the audio-from-video block's `parent_source_id` points to the video block's `source_id` (not the email) — preserves the derivation chain. Route uses `audioSourceId = videoSourceId + ":audio"` convention.
- **Shared helpers**: reuses `audioSegmentSchema` and `renderTranscript` from `audio.ts` (both exported) so transcript formatting is identical whether it came from an MP3 file or an MP4 track.
- Size cap from `PARSER_CONFIG.video.maxBytes` (50 MB). Oversize → `VideoTooLargeError`. Wrong type → `UnsupportedVideoTypeError`. Both are converted to `skipped` entries by the route. Route's `maxDuration` of 300s covers both video analysis + long audio transcription.

### Audio parser (universal, MP3 + M4A)

- `src/server/parsers/audio.ts` — `parseAudioBytes({ bytes, fileName, mediaType, … })` mirrors the PDF/image parsers: Gemini 2.5 Flash multimodal file part, universal across source systems. Transcribes + diarises + extracts structured metadata in a single call — no Whisper + second-pass LLM pipeline.
- **m4a support**: Google Chat voice messages (and Apple Voice Memos) ship as m4a — AAC audio inside an MP4 container. Callers can pass any reasonable mime spelling (`audio/mp4`, `audio/x-m4a`, `audio/m4a`, `audio/aac`, or even `application/octet-stream` with a `.m4a` filename); `resolveModelMediaType()` inside the parser normalises everything to `audio/mp4` (for m4a/aac variants) or `audio/mpeg` (for mp3) before the file part is sent to Gemini, which is fussy about the mime hint.
- **Schema includes a `segments[]` array** — each segment is `{ startSeconds, durationSeconds, speaker, transcript }`. Speaker labels are stable across segments (same voice = same label); the model uses a real name if the speaker introduces themselves or is addressed by name in the audio, otherwise `Speaker 1`, `Speaker 2`, …
- **Content markdown is assembled server-side** from `segments[]` in a compact format: `**00:15** · *7s* · **Speaker 2**\n\n<transcript>`. Top line shows total duration and speaker count. This keeps output format consistent regardless of what the model chose to emit as prose.
- Frontmatter: `senders = speakers` (they're the authors of the content), `recipients = []` (recordings don't have addressees by default; voicemail-style exceptions need a caller hint we don't have yet).
- Size cap from `PARSER_CONFIG.audio.maxBytes` (10 MB ≈ 10-15 min of 128 kbps mp3). Oversize → `AudioTooLargeError`. Wrong type → `UnsupportedAudioTypeError`. Both are converted to `skipped` entries by the route.
- Route's `maxDuration` is bumped to **300s** (5 min) for audio — transcription + diarisation is slower than image/PDF parsing, and Gemini's wall-clock varies with audio length.

### Image parser (universal, JPEG/PNG)

- `src/server/parsers/image.ts` — `parseImageBytes({ bytes, fileName, mediaType, … })` mirrors the PDF parser: Gemini 2.5 Flash multimodal file part (`{ type: "file", mediaType: "image/png" | "image/jpeg", data: bytes }`), single-pass, universal across source systems. Handles photo-style images (description) and text-heavy images (OCR) with one schema — the model picks the right strategy per image.
- Supported types come from `PARSER_CONFIG.image.supportedMediaTypes` (`image/jpeg`, `image/png`) + a filename-extension fallback (`.jpg`, `.jpeg`, `.png`) for providers that return `application/octet-stream`.
- Size cap from `PARSER_CONFIG.image.maxBytes` (5 MB). Oversize → `ImageTooLargeError`. Wrong type → `UnsupportedImageTypeError`. Both are converted to `skipped` entries by the route.

### PDF parser (universal)

- `src/server/parsers/pdf.ts` — `parsePdfBytes({ bytes, fileName, sourceId, parentSourceId, sourceSystem, threadId, sourceCreatedAt, sourceReceivedAt })` takes raw PDF bytes and produces a full parsed-source markdown block. Designed to be source-agnostic — the caller owns fetching and ID namespacing.
- Uses Gemini 2.5 Flash with a **multimodal file part** (`{ type: "file", mediaType: "application/pdf", data: bytes }`). The model "sees" the PDF visually, which lets it recognise and strip repeating headers/footers, page numbers, and watermark boilerplate while preserving heading hierarchy, tables, and lists. Single-pass — no chunking needed (Gemini Flash has 1M-token context).
- All frontmatter fields (`senders`, `recipients`, `mentions`, `companies`, `products`, `urls`) are extracted by the model — unlike emails, PDFs have no authoritative metadata envelope. `senders` = authors/signatories; `recipients` = explicit addressees (empty for reports/decks).
- Size cap from `PARSER_CONFIG.pdf.maxBytes`. Oversize files throw `PdfTooLargeError`, which the email route converts into a `{ skipped: "too large (…)" }` entry.

### Parse pipeline (`parseSourceItem`)

- **`src/server/parse-source-item.ts`** is the single entry point for parsing a synced `source_item`. Takes a row id, joins to `source` to read the provider, dispatches:
  - `nylas` → `parseEmailMessage(externalId)` → updates parent row with markdown; iterates `attachments[]` (real + CID-inline) and `bodyInlineImages[]` (data-URIs decoded server-side). Each attachment is downloaded via `nylas.attachments.downloadBytes`, classified by mime, parsed, written as a child `source_item` (parent-link to the parent row).
  - `gchat` → `parseChatMessage(externalId)` (full `spaces/X/messages/Y` path) → same shape; attachments downloaded via `downloadChatAttachmentBytes`.
  - `gdrive` → `parseDriveFile(externalId)` → first block updates parent; videos produce a second block (audio transcript) inserted as a child with `externalType = "derived_audio"`.
  - `dropoff` → throws. Drop-off rows arrive pre-parsed via `POST /api/sources/dropoff/upload` (the raw bytes aren't persisted, so there's nothing for the generic parse pipeline to re-run). The Processed table hides Re-parse on these rows; users re-upload to start over.
- **Error handling** is per-call, not per-batch:
  - The top-level call is wrapped: a 404/410 from the provider classifier (`isNylasItemMissing` / `isGoogleChatItemMissing` / `isGoogleDriveItemMissing` in `src/server/parsers/_provider-errors.ts`) marks the parent row `parseStatus = 'skipped'` with a "no longer exists at provider" message instead of `'failed'`. Anything else marks `'failed'` with the error message. **Either way the function returns normally** — critical for batch processing where one bad row mustn't crash the whole pass.
  - Each attachment download + parse is wrapped independently. A missing attachment becomes a child row with `parseStatus = 'skipped'`. A typed parser error (`PdfTooLargeError`, `UnsupportedAudioTypeError`, …) becomes `'skipped'` with the reason. An unexpected error becomes `'failed'`.
- **Re-parse** (`reparseSourceItem(itemId)`) deletes child rows (cascade picks up grandchildren via the `parent_source_item_id` ON DELETE CASCADE), then resets the root row's parse + R2 columns to defaults. The next Parse starts fresh. Roots only — children are recreated by their parent's re-parse.

### Analysis fields denormalised onto `source_item.metadata_json`

Two distinct things live in `metadata_json`:

1. **Sync-time provider fields** (subject, snippet, from/to for emails; author/text for chats; owners/createdTime for drive; etc.) — written by `src/server/sync/*.ts` at fetch time. Untouched by parse.
2. **Parse-time analysis fields** — denormalised copies of the LLM-extracted metadata that ALSO live in the markdown body's YAML frontmatter. Spread alongside (1) so `searchSourceItems` (chat tool) can `ILIKE` over `metadata_json::text` and match content that doesn't appear in the raw provider envelope (e.g., an email mentioning "Acme" only in the LLM-extracted `companies` list).

**⚠ Sync must MERGE, not overwrite `metadata_json`.** `upsertSourceItem` (`src/server/source-items.ts`) re-runs on every sync pass and only carries the sync-time fields (1). Its `onConflictDoUpdate` therefore sets `metadata_json = coalesce(existing,'{}') || <sync-fields>` (jsonb merge) — a plain overwrite would wipe the parse-time analysis (2) that `markParsed` merged in, since a re-sync of an already-parsed row would otherwise reset it to sync-only fields. The two key sets never collide, so the merge refreshes the envelope while preserving analysis. (This was a real bug: a second sync after parse silently emptied `companies`/`participantDetails`/etc., breaking discovery for the affected rows. `scripts/backfill-metadata-analysis.ts` re-extracts the searchable subset from R2 markdown for rows clobbered before the fix; a re-parse fully restores everything incl. `organizations`.)

**Analysis shape** (defined in [src/server/parsers/_shared.ts](src/server/parsers/_shared.ts) as `MetadataAnalysis`):

```ts
{
  language: string         // ISO 639-1
  summary: string          // 1-3 sentences (also rendered as ## Summary in body)
  mentions: string[]
  companies: string[]
  products: string[]
  relevance: { isJunk, category, reason }  // emails populate; everyone else gets DEFAULT_RELEVANCE
}
```

**Important — `contentMarkdown` is NOT denormalised.** It stays only in the markdown body / R2 blob. Putting it in `metadata_json` would bloat the column for no search benefit (the body is too long to ILIKE usefully).

**How it's written**:
- Each parser returns an `analysis: MetadataAnalysis` field alongside its existing `markdown` + `metadata` (video returns two — `videoAnalysis` + `audioAnalysis` — for its dual blocks; drive's per-block array carries one per block).
- `markParsed(itemId, markdown, analysis)` in `src/server/parse-source-item.ts` merges via Postgres jsonb `||`: `metadata_json = COALESCE(metadata_json, '{}'::jsonb) || $analysis::jsonb`. Sync fields preserved, analysis keys take precedence on collision (none today).
- `insertParsedChild({ ..., analysis })` writes the analysis as the entire `metadata_json` for new child rows (children are synthesised — they have no provider sync metadata to preserve).
- The two upload routes that bypass `parseSourceItem` and insert pre-parsed rows directly — `POST /api/sources/dropoff/upload` and `POST /api/sources/whatsapp/upload` — spread `firstBlock.analysis` / `extra.analysis` into `metadata_json` at insert time so files arrive with analysis populated.

**Backfill script** — [scripts/backfill-metadata-analysis.ts](scripts/backfill-metadata-analysis.ts) populates the analysis fields on rows parsed before this denormalisation existed. Strategy: NO LLM, NO provider re-fetch — re-extracts from the markdown the parsers already wrote.

- **Source of markdown** per row: `parsed_markdown` column first, fall back to `getMarkdownFromR2(markdownR2Key)` for already-uploaded rows.
- **Extraction**: small inline YAML reader pulls `language`, `mentions`, `companies`, `products` out of the frontmatter block (handles quoted scalars with escaped chars, `key: []` empty form, multi-item lists). `summary` is grabbed from the `## Summary` section body.
- **`relevance` defaults** to `{ isJunk: false, category: null, reason: "" }` — junk emails are kept as `parse_status='skipped'` with no markdown to backfill from, so non-skipped rows are by definition non-junk.
- **Idempotency**: skips rows whose `metadata_json.summary` is already a non-empty string. `--force` overrides.
- **Self-contained**: the script inlines the small bits it needs from `_shared.ts` and `r2.ts` because both files start with `import "server-only"` (a Next.js virtual module that crashes under tsx).

Usage:
```sh
pnpm tsx scripts/backfill-metadata-analysis.ts                     # dry run (default)
pnpm tsx scripts/backfill-metadata-analysis.ts --apply             # persist
pnpm tsx scripts/backfill-metadata-analysis.ts --apply --limit=50  # batch
pnpm tsx scripts/backfill-metadata-analysis.ts --apply --force     # re-extract over existing analysis
```

Reports `scanned / alreadyDone / noMarkdown / malformed / updated / failed` counters at the end. Logs `[backfill] <id>: <reason>` for any row that's skipped or failed so you can grep for outliers.

### Endpoints (Phase 2)

All routes below are **org-scoped** (not platform-admin-gated) — they require an authenticated session with an active organization, and per-row mutations call `assertSourceItemInScope` / `assertSourceInScope` to verify the target row belongs to the caller's active org. 401 = no session / no active org; 403 = scope mismatch; 404 = not found.

- `POST /api/sources/items/[id]/parse` — `maxDuration = 300` (videos). Body: none. Returns `{ parentStatus, parentMarkdownBytes, childInserted, childSkipped, childFailed }`.
- `POST /api/sources/items/[id]/process` — `maxDuration = 300`. The unified one-shot used by the "Источники организации" tab: `processSourceItem(id)` (in `src/server/process-source-item.ts`) parses (only if not already complete/skipped) → uploads the item → uploads any child rows a fresh root parse just produced. Non-throwing (mirrors `parseSourceItem`/`uploadSourceItem`); returns `{ ok, parseStatus, r2UploadStatus, parsed, uploaded, childrenUploaded, childrenFailed, error? }`. Idempotent/resumable: a parsed-but-unshipped row only retries the upload (no wasted LLM). Re-uses the exact building blocks the daily cron uses, so manual and cron paths can't drift.
- `GET /api/sources/items/process-ids` — work-set ids (parse pending/failed OR parsed-but-unshipped) in the table's filter context, hard-capped at 500. Powers the "Обработать все (N)" count + run. Mirrors `pending-parse-ids` but spans both sub-processes (`listProcessItemIds`).
- `POST /api/sources/items/[id]/reparse` — Body: none. Returns `{ ok: true }`.
- `GET /api/sources/items/[id]/markdown` — Returns `{ markdown, source: 'db' | 'r2' }` for parsed rows. Used by the Show modal. Reads `parsed_markdown` first; falls back to `getMarkdownFromR2(markdownR2Key)` for already-uploaded rows.
- `POST /api/sources/r2/save` — Body: `{ sourceItemId }`. Loads the row's `parsed_markdown`, uploads to R2 under the canonical key `org_<orgId>/source_<sourceId>/item_<itemId>.md`, stamps `r2UploadStatus = 'complete'` + `r2UploadedAt` + `markdownR2Key` + `markdownR2SizeBytes`, **clears `parsed_markdown`** (R2 has the canonical copy now). On R2 failure: `r2UploadStatus = 'failed'`, `parsed_markdown` left intact for retry. The route is a thin org-scope check around `uploadSourceItem()` in `src/server/r2/upload-source-item.ts`; the daily orchestration calls the same function directly without going through HTTP.
- `POST /api/sources/dropoff/upload` — `maxDuration = 300`. Multipart body with `files[]` + optional `description`. Looks up the dropoff source for the caller's active org (`provider = 'dropoff' AND owner_organization_id = activeOrgId AND status = 'active'`); 404 if no drop-off source is configured. For each file: runs `parseDropoffFile` inline and inserts a `source_item` root (+ video-audio child where applicable) with `parseStatus = 'complete'` and `parsed_markdown` populated. Returns `{ results: { fileName, ok, error?, reason? }[], okCount, failedCount }`. Per-file failures (unsupported / oversize / parser error) don't roll back the batch — successful siblings still land.
- `GET /api/sources/org` and `PATCH /api/sources/org` — **org-owner-only** management surface for the "Manage organization sources" tab. GET returns `{ sources: OwnerOrgSource[] }` for the active org; PATCH body `{ sourceId, automatedParsingIsAllowed?, status? }` flips one of the two editable knobs. Owner check is enforced server-side via `requireOrgOwner()` in `src/server/sources.ts`; non-owners get 403. See § "Org-owner sources management" above.
- `POST /api/sources/whatsapp/upload` — `maxDuration = 300`. Multipart body with `files[]` + parallel `paths[]` (each file's `webkitRelativePath`). Looks up the WhatsApp Archive source for the active org. Splits work by filename: `_chat.txt` (case-insensitive) → parsed via `parseChatTxt` into time-bounded chat groups inserted as `chat_message` source_items with `parseStatus = 'pending'` and the rendered transcript persisted in `metadataJson.rawText`. Every other file → parsed inline (PDF / image / audio / video / Office) and inserted as `attachment` source_items with `parseStatus = 'complete'`. Filename-matched attachments get `parent_source_item_id` linked to the claiming chat group. `claimedBy` map is built from BOTH the current request's groups AND existing chat-group rows in the DB, so chunked uploads work correctly across multiple POSTs (chat in chunk 0, media in chunks 1+). Dedup via content-hash external ids (`whatsapp-att:<hash>`) and group-key external ids (`whatsapp:group:<hash>`). Returns `{ chatFormat, groupCount, attachmentCount, failedCount, results }`. See § "WhatsApp Archive" below.
- `GET /api/sources/items/pending-parse-ids` — returns `{ ids, total, cap }` for root rows in `parseStatus IN ('pending', 'failed')` matching the same filter context as the Pending listing (sourceId, q, date_from, date_to, scope). Hard-cap 500 per call. Powers the "Parse all" batch control on Pending.
- `GET /api/sources/r2/pending-ids` — returns `{ ids, total, cap }` for rows ready for R2 upload (`parseStatus = 'complete' AND r2UploadStatus IN (pending, failed) AND parsed_markdown IS NOT NULL`). Filter context mirrors the Processed and Stored Content tables (sourceId, q, mime, date_from, date_to, scope). Hard-cap 500. Powers the "Upload all to R2" batch control.

### UI: Unified source-items table

- `src/components/tables/table-source-items.tsx` — the **single unified table** (roots + children) for the org-sources tab. Columns: Дата / Источник / Автор / Элемент / **Разбор** / **Загрузка** / Действия. Two independent badges show each sub-process; a smart per-row **«Обработать»** button does whatever work remains (`POST /process`), plus **Show** (when `parseStatus = 'complete'`) and **Re-parse** (roots whose provider `hasRawBytesPersisted`). `skipped` rows show no action (terminal). Children render with a `↳` prefix; they only ever need an upload, which the same «Обработать» button covers. Per-row buttons are disabled while a shared batch run (`processRunning`) is in flight.
- **Row state → action**: needs work = `parseStatus ∈ {pending,failed}` OR (`complete` & `r2 ≠ complete`) → «Обработать» (label «Повторить» when something failed); `processing` → inline spinner; `complete`+uploaded → Show (+ Re-parse); `skipped` → «—».
- **`itemTitle` helper** maps each `externalType` to a display string. For `chat_message` rows it tries `metadataJson.text` first (Google Chat single-message convention), then falls back to `previewFromRawText(metadataJson.rawText)` for WhatsApp groups (skips the `**TIMESTAMP** · **AUTHOR**` header lines + `_attachments:_` markers, returns the first body line). Without this fallback every WhatsApp row renders as "(empty message)".
- `src/components/blocks/parsed-markdown-dialog.tsx` — Show modal. `sm:max-w-5xl` width, single `overflow-y-auto` scroll container at the dialog level. Fetches `/api/sources/items/[id]/markdown` and renders via `<ParsedBlock>` (one card: YAML frontmatter + Summary + Content + a small `Copy` toolbar).
- After any per-row action the table bumps a local refresh + calls `onActionComplete()` (→ scope `refreshKey`), so the row's badges flip in place and the «Обработать все» count re-polls.
- **Failure surfacing**: `processSourceItem` returns HTTP 200 with `ok:false` + `error` on partial failure (batch-safety). The per-row handler and the `useProcessRun` loop both check `ok === false` and convert it to a red error toast — a silent LLM rate-limit failure never masquerades as success.

### Batch controls (top-of-table)

Batch operations live in self-contained components that mount above the filter row of their respective tables. Each follows the same pattern: idle "Action all (N)" button + filter-context-driven count, running progress bar with cancel, server endpoint that returns IDs in current scope, client loop with parallel chunks. Hidden when `N = 0`.

- **Unified process run — `useProcessRun` + `<ProcessAllButton>` + `<ProcessRunBar>`** (`src/components/blocks/use-process-run.ts`, `process-controls.tsx`). Replaced the old separate `<ParseBatchControl>` (deleted) on the org-sources tab. `useProcessRun({ onRefresh })` owns ONE run at a time (cancel, progress, toasts) and is shared by the table's "Обработать все" button (`<ProcessAllButton>`, which polls `/process-ids` for the count in the current filter context) AND the per-source sync buttons (`onProcessSource`). The loop pulls work-set ids then calls `POST /process` per item in chunks. Concurrency: default 3-wide / no delay; **slow mode** (checkbox) 1-wide + 4s gap for AI-Gateway free-tier limits — read at click-time. Cancel checked at chunk boundaries; in-flight `/process` calls finish naturally. 500/click cap with a "run again" toast. Per-item failures aggregated into toasts (cap 8 + "+N more"); failed rows keep their red badge for retry. Crash-safe: each `/process` is one bounded request and writes durable status, so a closed tab / mid-run cancel just resumes on the next run.
- **`<R2BatchUploadControl>`** (`src/components/blocks/r2-batch-upload-control.tsx`) — **now only mounted on the Stored Content table** (the org-sources tab uses the unified process run above). Targets `parseStatus = 'complete' AND r2UploadStatus IN (pending, failed) AND parsed_markdown IS NOT NULL`. Concurrency: 5 parallel R2 puts. Same cancel + filter-context conventions.
- **Filter scope is filter-driven, not selection-driven**: both controls read the host table's current filter state (sourceId / q / date / mime / scope) and pass it to their `pending-ids` endpoint. So filtering Processed to "Source = WhatsApp Archive" and clicking "Upload all to R2" ships exactly the visible scope. No row-level checkboxes — keeps the UI simple and the operator mental model consistent ("filter, then act").
- **Hard cap**: 500 per click on each control. If `total > 500` the trailing toast tells the user to re-run for the next batch. Repeat clicks dedup naturally because already-processed rows leave the candidate set.
- **Live refresh**: each chunk completion calls `onActionComplete` which bumps the host table's `refreshKey`. Per-row badges flip from "Needs parse"/"Needs upload" → "Done" while the batch is still running.
- **Failure aggregation**: per-failure toasts cap at 8 + a "+N more" line so a catastrophic batch doesn't bury everything else.

### Processed table — "Needs upload only" filter

Checkbox in the Processed filter row (next to the date-preset buttons). When checked, the listing query narrows from `parseStatus IN (complete, skipped)` to `parseStatus = 'complete' AND r2UploadStatus IN (pending, failed)` — drops both finished-and-uploaded rows AND skipped rows (which have no markdown to ship). Useful for working through the upload backlog without scrolling past finished items. Server flag: `?only_needs_upload=1` on `/api/sources/items`. Counts as an active filter for the "Clear filters" button's enabled-state.

### Save to R2

- **Goal**: upload a parsed `source_item`'s markdown to the Cloudflare R2 bucket configured in `.env`, and stamp the upload-side columns on the row.
- **Client**: `src/lib/r2.ts` — thin `S3Client` wrapper using `@aws-sdk/client-s3` (R2 is S3-compatible per Cloudflare docs). Reads `R2_PUBLIC_URL` as the S3 endpoint (or constructs `https://<R2_ACCOUNT_ID>.r2.cloudflarestorage.com` if unset). Exports `putMarkdownToR2(key, markdown)` and `getMarkdownFromR2(key)`. The client is lazy and only throws on missing creds when first used so the app boots without R2.
- **API**: `POST /api/sources/r2/save` (org-scoped, `maxDuration = 30`). Body: `{ sourceItemId }`. Verifies the item belongs to the caller's active org via `assertSourceItemInScope`, then calls `uploadSourceItem()`. The markdown comes from `source_item.parsed_markdown` (set by Parse). Key follows the canonical schema convention `org_<orgId>/source_<sourceId>/item_<itemId>.md`. On success: `r2UploadStatus = 'complete'`, `r2UploadedAt`, `markdownR2Key`, `markdownR2SizeBytes` are stamped, and `parsed_markdown` is **cleared** (R2 has the canonical copy now; the Show modal falls back to fetching from R2). On R2 failure: `r2UploadStatus = 'failed'`, `parsed_markdown` left intact for retry.
- **UI**: per-row `Upload` button on Processed table rows where `r2UploadStatus !== 'complete'`. The legacy per-`<ParsedBlock>` "Save to R2" button (`sourceId` + `parentSourceId` props) is no longer used by any caller after the panels were removed in Phase 2; the props remain on the component for any future ad-hoc rendering.

### Source dictionary integration

  - `listOrgSources(orgId)` — `owner_organization_id = orgId AND status = 'active'`. Drives the "Your organization sources" tab's action bar + the per-table source filter.
  - `listOwnerOrgSources()` — same scope as `listOrgSources` but goes through `requireOrgOwner()` and returns the wider `OwnerOrgSource = { id, provider, name, description, status, automatedParsingIsAllowed, lastSyncedAt }` shape used by the owner-side management table.
- `provider_config` payloads (initially filled by `scripts/seed-system-sources.ts`, copied to per-org rows during the org-scoping migration; re-runs of the seed script only patch rows whose config is still `{}`):
  - `Emails` (nylas) → `{ grantId: NYLAS_GRANT_ID }`
  - `Google Chat` (gchat) → `{ spaceId: "spaces/AAQA8JhMLes" }`
  - `Google Meet` (gdrive) → `{ driveId: "0ADuGKT0uyTURUk9PVA" }`
  - `Google Drive` (gdrive) → `{ driveId: "0ALbtkFVF6pVEUk9PVA" }`
  - `Files Drop Off` (dropoff) → `{}`
  - `WhatsApp Archive` (whatsapp) → `{}` — seeded per-org via `scripts/seed-whatsapp-source.ts --org=<orgId>`. Not part of the system-sources seed because it's strictly per-org from day one.
- **Why split provider_config vs credentials**: `provider_config` holds non-secret connection routing (`spaceId`, `driveId`); `credentials_ref` holds the per-source secret payload, encrypted. The migration script that seeded the existing rows also strips `grantId` out of nylas `provider_config` — it was a leak of secrets into a non-secret column. Going forward, the credentials form is the only path that writes secrets, and they always land encrypted.
- **Phase 2 update — sources via templates**: every `source` row now carries an optional `template_id` linking back to the `source_template` it was instantiated from. The previous "rename `scripts/seed-system-sources.ts`" follow-up is partially obsolete — new orgs auto-bootstrap from `is_default = true` templates via the org-creation hook. The seed-system-sources script is still around for cleanup work tracked separately.
- **Note**: `scripts/seed-system-sources.ts` still seeds `is_system = true` rows on re-run. This wouldn't undo the existing per-org rows, but it would re-create the now-empty System bucket. Worth refactoring to a per-org seeder before the next clean install; tracking but not blocking.
- The `/sources` page reads source lists server-side and threads each `source.id` through to the sync + parse pipelines. After Phase 2, no part of the page reads from the providers directly — the unified Pending/Processed tables are entirely DB-backed. The legacy `/api/{emails,chats,drive}` list endpoints and `/api/sources/parse/{email,chat,drive,dropoff}` parse endpoints have been removed; only the attachment-download proxies remain (`/api/emails/attachments`, `/api/chats/attachments`, `/api/drive/download`) for the Show-original side flow.

`src/server/parsers/{audio,video,docx,xlsx,…}.ts` will slot in as peers. ## Provider integrations (sync side)

The provider-list endpoints (`/api/emails`, `/api/chats`, `/api/drive`) and the provider-specific parse routes are **gone**. All sync now flows through `POST /api/sources/sync` → `src/server/sync/{nylas,gchat,gdrive}.ts`, which reads each row's `provider_config` for the connection identifier and uses the existing SDK clients in `src/lib/{nylas,google-chat,google-drive}.ts`. The attachment download helpers stay since both the parser pipeline and the on-row "view original" flow rely on them:

- **Emails attachments** (`src/app/api/emails/attachments/route.ts`) — downloads via `nylas.attachments.downloadBytes()`; requires `attachmentId` and `messageId`.
- **Chat attachments** (`src/app/api/chats/attachments/route.ts` + `downloadChatAttachmentBytes()` in `src/lib/google-chat.ts`) — `media.download` is user-auth-only, so the route impersonates `GOOGLE_CHAT_IMPERSONATE_USER` via DWD with `chat.messages.readonly` and hits `https://chat.googleapis.com/v1/media/{resourceName}?alt=media` directly.
- **Drive download proxy** (`src/app/api/drive/download/route.ts`) — serves file bytes with `Content-Disposition: attachment` so the browser triggers a download rather than rendering inline. Branches:
  - Google Docs → `files.export` as `text/markdown` (`.md`).
  - Google Sheets → `files.export` as `.xlsx`.
  - Google Slides / Drawings → `files.export` as PDF.
  - Folders / Forms / Shortcuts → explicit 400.
  - Everything else → `files.get({ alt: "media", supportsAllDrives: true })`.

**Auth requirements** for the Google integrations:
- Chat list: `chat.app.messages.readonly` on the service account client ID (Workspace Admin → Security → API controls → Domain-wide delegation).
- Chat attachment download: same client ID must also have `https://www.googleapis.com/auth/chat.messages.readonly`, and `GOOGLE_CHAT_IMPERSONATE_USER` must have access to the space.

## WhatsApp Archive (Sources)

Local-archive import for WhatsApp chat exports. The user picks a folder via the browser's directory picker; the dialog uploads the contents in chunks; the server splits the load into chat groups (text history) and attachments (media), inserts both as `source_item` rows, and links attachments to the chat group that referenced them.

- **Provider**: `whatsapp` in the `sourceProvider` enum. One source per org (`is_system = false`, `owner_organization_id = <orgId>`). Seeded via `scripts/seed-whatsapp-source.ts --org=<orgId>`. `provider_config = {}`; the source has nothing to fetch from a remote API. `automatedParsingIsAllowed = false` by default — the daily cron has nothing to do for WhatsApp, so leaving auto-parse on would just waste cycles.
- **Parser** (`src/server/parsers/whatsapp.ts`):
  - `parseChatTxt(content)` handles iOS (`[m/d/yy, hh:mm:ss AM/PM] Author: text`) and Android (`m/d/yy, hh:mm - Author: text`) export formats. Date separator is `[/.]` — covers US `/` exports and EU `.` exports both. Auto-detects mm/dd vs dd/mm by checking if any month component > 12 across a 200-line sample. Strips bidi marks (LRM / U+200E, U+200F, BOM) before regex matching. Folds multi-line continuations into the previous message. Extracts `<attached: filename>` markers (and Android's `filename (file attached)` convention) into a per-message attachment list.
  - **Group splitter**: starts a new group when the gap from the previous message exceeds 1 hour, OR when the bucket already has 100 messages, OR when the bucket exceeds 50 000 chars of body text. Cap protects the LLM from one 8-hour active conversation becoming a single 200-message blob.
  - `groupKey = sha256(firstISO::lastISO::authorsJoined).slice(0, 16)` — stable across re-imports for unchanged groups (idempotent dedup).
  - `renderGroupTranscript(group)` produces a markdown transcript like `**2025-12-09 12:57** · **Author**\n\nbody text\n\n_attachments: \`file.jpg\`_`. Persisted in `metadataJson.rawText` at upload time and re-used at parse time — WhatsApp has no remote API to re-fetch from, so the group's content lives in our DB.
  - `parseWhatsAppGroup({ rawText, sourceId, threadId, authors, startTimestamp, endTimestamp })` — LLM metadata pass mirroring the Google Chat parser. Schema: `{ language, summary, mentions, companies, products }`. Content markdown is the original transcript verbatim — no LLM rewrite, just metadata extraction.
- **Upload route** (`POST /api/sources/whatsapp/upload`, `maxDuration = 300`):
  - Multipart form: `files[]` + parallel `paths[]` (`webkitRelativePath` per file, sent in same order so server can pair them).
  - Locates `_chat.txt` case-insensitively from the path's last segment.
  - **Pass 1 — chat groups**: parses `_chat.txt` if present, inserts each group as a `chat_message` source_item with `parseStatus = 'pending'` and `metadataJson = { provider, format, groupKey, startTimestamp, endTimestamp, authors, messageCount, attachmentFilenames, rawText }`. `filename` left null (the group transcript is multi-message; conflating with a file confuses the Stored Content "Filename" column).
  - **Pass 2 — `claimedBy` map**: filename → chat-group row id. Built from BOTH the current request's groups AND existing `chat_message` rows in the DB for this source. The DB lookup is what makes chunked uploads work — chunk 0 carries `_chat.txt` and inserts groups; chunks 1+ carry only media but still find their parent via the DB pre-fill. First-reference-wins for duplicate filenames.
  - **Pass 3 — attachments**: each non-chat file → `parseDropoffFile` (reuses every existing per-format parser + size cap + skip semantics). Inserts as `attachment` source_item with `parseStatus = 'complete'` and `parsed_markdown` populated. `parent_source_item_id = claimedBy.get(filename) ?? null`. Video → derived audio block becomes a child of the video row, mirroring dropoff convention.
  - **Dedup**: chat groups keyed `whatsapp:group:<groupHash>`; attachments keyed `whatsapp-att:<contentHash>` (sha256 of bytes, first 16 hex chars). Re-uploading the same archive → existing rows skipped. Re-exporting after new chat activity → only new groups insert.
  - Per-file failures (unsupported / oversize / parser error) don't roll back the batch — successful siblings still land. Returns `{ chatFormat, groupCount, attachmentCount, failedCount, results }`.
- **Parse pipeline integration** (`src/server/parse-source-item.ts`):
  - `ParseContext` carries `metadataJson` (snapshot of `source_item.metadata_json`) so providers without remote APIs can read their content from the row.
  - `parseWhatsAppItem(ctx)` reads `rawText` + `authors` + timestamps from the metadata, calls `parseWhatsAppGroup`, marks the row complete with the assembled markdown. No child rows produced here — attachment children were inserted at upload time.
  - `parseSourceItem` returns `parentParseError` in addition to `parentStatus` when failed/skipped, so HTTP callers can show meaningful error toasts without a follow-up DB read.
- **Dialog** (`src/components/blocks/whatsapp-archive-dialog.tsx`):
  - Folder picker via `<input type="file" webkitdirectory directory multiple>` — universal browser support (Chrome / Firefox / Safari / Edge).
  - Filters out OS junk (`.DS_Store`, `Thumbs.db`).
  - Shows file count + total size + a green "Found `_chat.txt`" badge or amber "media-only import" warning before submit.
  - **Chunked upload**: `CHUNK_SIZE = 5`. `_chat.txt` always lands in its own dedicated first chunk so chat groups exist before any media chunk needs parent linkage. Per-chunk progress: bar fills as files are attempted (monotonic), with running counters for groups / attachments / failed.
  - Cancel button is intentionally absent on this dialog — chunks are independent and short, so just closing the dialog mid-batch is safe (already-uploaded chunks stay in the DB; remaining chunks just don't run).
  - Per-file failure toasts cap at 8 + a "+N more" line.
- **Action bar button** (`<SyncActionBar>`): "Sync from WhatsApp Archive" appears on the right when the org has a `whatsapp`-provider source. Filters whatsapp out of the per-source Sync button row (the dedicated button replaces it).
- **Back-link script** (`scripts/whatsapp-backlink-attachments.ts`): one-shot, idempotent. Iterates orphan `attachment` rows (`parent_source_item_id IS NULL`) under the org's WhatsApp source and pairs them with chat-group rows by filename via `metadataJson.attachmentFilenames`. Used once after the initial broken-regex import to retroactively link attachments that landed before the chat groups existed. Never overwrites an existing link. Re-runnable safely.

## AI Chat (Sources)

Internal-type source for persisting AI-chat sessions from the `/dashboard` chat panel. Triggered by a **Save Chat** button in the chat header — last button in the row, only visible when `messages.length > 0`. Any org member can use it.

- **Schema**:
  - `sourceProvider` enum gained `aichat`. `sourceItemKind` gained `aichat_session` (root row per saved session). Children that come from file attachments use the existing `attachment` kind for symmetry with dropoff/whatsapp.
  - `source.type` is set to `"internal"` for AI Chat (only source today using that value — every other seeded source is `"external"`). The `type` enum has been there all along; we just hadn't had a use for `"internal"` until this feature.
- **Lazy provisioning** (`src/server/sources.ts → getOrCreateAiChatSource(orgId)`): looks up the org's `(provider='aichat', ownerOrganizationId=orgId)` source row; inserts on first miss with the spec defaults (`type: 'internal'`, `name: 'AI Chat'`, `description: 'Save AI-Chat data to the system context'`, `status: 'active'`, `isSystem: false`, `automatedParsingIsAllowed: false`, empty `providerConfig`, no `credentialsRef`). Idempotent — subsequent saves find the existing row. **No seed script needed** — the source materialises on first save per org.
- **Save flow** (`src/server/save-chat-session.ts → saveChatSession()`):
  1. **Conversation extraction** — the dialog walks `useChat`'s `UIMessage[]` client-side, keeps only `text` parts (per-role) and `file` parts whose `url` is a `data:` URL. Tool calls / reasoning / json-render specs / Google-grounding source URLs are dropped (Q4 — conversation only). Files referenced by remote URL or whose data URL has been GC'd by the SDK are counted as `droppedFileCount` and surfaced in the modal as a warning.
  2. **Markdown rendering** — `### User` / `### Assistant` blocks in chronological order, plus a `### Files attached` section listing filenames at the end.
  3. **Analysis** — single Gemini 2.5 Flash call with `Output.object({ schema })` extracting `language / summary / mentions / companies / products`. `relevance` defaults to `DEFAULT_RELEVANCE` (chat sessions aren't junk-classified). Failure here aborts the save with a toast — no half-saved row.
  4. **Parent insert** — `source_item` row with `externalType: 'aichat_session'`, `externalId: 'aichat:<uuid>'`, `parseStatus: 'complete'`, `parsedMarkdown: <transcript>`, `metadataJson: { provider, title, messageCount, attachmentCount, savedByUserId, savedAt, ...analysis }` (the analysis fields land top-level, same as every other source's denormalisation pattern).
  5. **Per-file children** — each extracted file goes through `parseDropoffFile()` (reusing the entire dropoff parser dispatch — PDF / image / audio / video / office). Inserts as `attachment` child with `parent_source_item_id = <parent>`. Video → audio derivation lands as a `derived_audio` grandchild, mirroring the dropoff convention. Per-file errors are classified as `skipped` (size / unsupported-type) or `failed` (anything else) and become placeholder rows so the operator sees what didn't land.
- **API**: `POST /api/sources/aichat/save`, `maxDuration = 60`. Multipart body — `title` (string), `messages` (JSON-stringified `ChatMessageInput[]`), `files[]` (binary parts, in order). Hard cap of 50 MB total upload to prevent memory blow-out (per-file parser caps still apply on top). Returns `{ itemId, parentExternalId, childInserted, childSkipped, childFailed, perFileResults }`.
- **UI** (`src/components/blocks/save-chat-dialog.tsx`):
  - Triggered by the labeled `<Save> Save Chat` button in the chat header (next to and AFTER `Clear conversation` — last in row, conditional on `messages.length > 0`).
  - Modal shows a Title field pre-filled from the first user message (capped ~60 chars; user edits before confirming), plus a summary line ("Will save N messages and M files") and an amber warning when any attachments couldn't be recovered.
  - Save submits the multipart form. On success: toast (`Chat saved to sources · N files attached`), close. **Does not clear the chat** (Q2 — non-destructive). User uses the existing trash icon to start fresh.
- **No re-parse / no sync**:
  - `parseSourceItem(itemId)` throws for `aichat` provider — sessions are always `parseStatus='complete'` at insert time; the unified parse pipeline has nothing to re-run.
  - `syncSource(sourceId)` throws for `aichat` — there's no remote provider to fetch from. The Sources-page action bar's per-source Sync buttons skip aichat naturally (same as dropoff/whatsapp).
- **Sources page integration**: AI Chat rows show up in the standard Pending/Processed tables under `/sources` once parsing lands the row directly into Processed. They participate in the same R2 upload flow as every other source (manual click of the per-row Upload button or the "Upload all to R2" batch control).

## Telegram Bot (Sources)

Per-org Telegram bot ingestion. **Phase 1 (DM text) is DONE + verified in production.** Each org's admin creates their own bot via @BotFather and stores the token on the source — the bot **is** the tenant, so there is **no shared bot and no deep-link tenant binding** (a deliberate divergence from `refs/telegram-bot-ingestion-spec.md` §B5). Built on **grammY**, not the Vercel Chat SDK. Push-ingested via webhook (no remote sync/pull). Future work (attachments, groups, hardening) is `PHASE2.md` #20.

- **Provider**: `telegram` in the `sourceProvider` enum. `is_system = false`, `owner_organization_id = <orgId>`. Seeded as a template via `scripts/seed-templates.ts` (`Telegram Bot`, `is_visible_to_orgs = true`, `is_default = false`, `default_automated_parsing_is_allowed = true`). Registry capabilities: `supportsRemoteSync = false` (NOT in `runFullSync()` cron — Telegram is push), `supportsManualFetch = true`, `supportsAutomatedPipeline = true`, `hasRawBytesPersisted = true` (Re-parse reads `metadata_json.rawText`).
- **Credentials** (`credentials_ref`, schema in `src/server/providers/handlers.ts`): `{ botToken, webhookSecret }`. `botToken` = BotFather token; `webhookSecret` = high-entropy string the operator generates, echoed by Telegram in `X-Telegram-Bot-Api-Secret-Token`. `getTelegramCredentials` (`src/server/providers/credentials.ts`) decrypts/validates, with a **nylas-style env fallback** to `TELEGRAM_BOT_TOKEN` + `TELEGRAM_WEBHOOK_SECRET_TOKEN` (both required) when `credentials_ref` is null. **Non-secret** `provider_config`: `{ botUsername?, telegramOffset? }` (`botUsername` optional, for Phase 3 mention detection; `telegramOffset` is the manual-fetch cursor).
- **Webhook auto-registration**: saving credentials (owner `updateOwnerOrgSourceCredentials` or admin `updateAdminSourceCredentials`) fires `registerTelegramWebhookForSource` → Telegram `setWebhook` at `${NEXT_PUBLIC_APP_URL}/api/webhooks/telegram/<sourceId>` with the per-source secret (`allowed_updates: ["message"]`, `drop_pending_updates: true`). **Skipped (best-effort, logged) when `NEXT_PUBLIC_APP_URL` is not a public HTTPS origin** (local dev) — then register manually, but the URL MUST include the `/<sourceId>` path segment or every delivery 404s. Diagnose with Telegram `getWebhookInfo` (`url` / `pending_update_count` / `last_error_message`).
- **Webhook route** (`src/app/api/webhooks/telegram/[sourceId]/route.ts`, **public, org-unscoped**, Node runtime, `force-dynamic`, fast-ack): outside `(protected)`; authorization = path `sourceId` + secret header (mirrors the guest order link). `POST` → `resolveTelegramWebhookContext(sourceId)` (loads source→org strictly by path id; 404 if not an active telegram source; 503 if creds unconfigured so Telegram retries) → `ingestTelegramUpdate` verifies the secret (constant-time), persists, acks. Errors → 401 (bad secret) / 404 / 503 / 500.
- **Ingestion** (`src/server/ingest/telegram.ts`): `persistTelegramMessage` is the shared core (no secret, no ack) — **Phase 1 scope = private-chat TEXT only** (groups Phase 3, attachments Phase 2). Writes a `chat_message` source_item, `external_id = <chat_id>:<message_id>` (idempotent via `UNIQUE(source_id, external_id)`), `metadata_json = { provider, rawText, text, senders, telegram: { updateId, messageId, chatId, chatType, date, from } }`, `parse_status = 'pending'`. The webhook path adds secret-verify + a best-effort `✅ Received` ack (ack failure never fails ingestion). Then the **normal parse → R2 pipeline** picks it up (no special-casing downstream).
- **Parser** (`src/server/parsers/telegram.ts` → `parseTelegramMessage`): WhatsApp-style — reads the body from `metadata_json.rawText` (no remote re-fetch), one Gemini metadata pass (`language/summary/mentions/companies/products/mentionedPeople` + clean `contentMarkdown`), assembles template markdown with `sourceSystem = "Telegram"`. Wired into `parseSourceItem` via `parseTelegramItem(ctx)` + the `telegram` arms of the provider union / namespaced-id (`telegram:<chat>:<msg>`) / source-label switches.
- **Manual fetch** (`POST /api/sources/telegram/fetch`, org-scoped: `getServerSession` + `assertSourceInScope`): the `<TelegramFetchButton>` on `<SyncActionBar>` ("Fetch {name}"). `fetchTelegramUpdates(sourceId)` drains `getUpdates` (up to 1000/click), persists via the same `persistTelegramMessage`, and advances the `provider_config.telegramOffset` cursor. **Mutually exclusive with the webhook**: while a webhook is active Telegram 409s, surfaced as `{ webhookActive: true }` → an info toast ("messages arrive automatically"). Primarily a local-dev tool (no webhook reachable).
- **Operational notes** (learned in prod bring-up): set `NEXT_PUBLIC_APP_URL` to the HTTPS origin on Vercel so auto-registration works; ingested rows are org-scoped to the source's owner org, so a "webhook green but nothing visible" symptom is usually the wrong **active org** in the app. See `PHASE2.md` #20 for the full deferred list (attachments, media-groups, groups, rate-limit, re-register button, webhook-health surfacing).

## Clients & Contacts

- **Schema** (`src/db/schema.ts`): `client` + `contact` tables. `funnel_phase` enum (awareness/interest/decision/action/retention). `entity_status` enum (`active` | `suspended` | `initial` | `deleted`) acts as **soft-delete + review-queue** — records are never hard-deleted; `suspended` hides a manually-archived row, `initial` flags an auto-discovered row that the operator hasn't reviewed yet (see § "Client discovery from sources" below), and **`deleted` is a soft-delete for test artifacts / mistakes** — hidden from the Clients & Contacts lists by default (the Status filter's "All statuses" excludes it; pick `deleted` explicitly to view/restore). **A `deleted` client/contact is treated as ABSENT by discovery and is REVIVED on re-scan (not re-created as a duplicate).** This powers the rules/parsing/attribution test cycle: soft-`delete` a record, re-run "Discover from sources", and it resurfaces as a candidate; on apply, `applyDiscovery` flips the SAME soft-deleted row back to `status='initial'` (reusing its id, so deals / cards / contact links that referenced it stay intact) rather than inserting a new row. ### Unified discovery from sources

One **"Discover from sources"** button on `/clients` (Clients & Contacts tab toolbar, alongside "New client" / "New contact") opens a single `<DiscoverDialog>` that runs one scan and surfaces three candidate sets together — **Companies**, **Contacts**, and **Links** — replacing the three legacy flows (Discover clients / Discover contacts / Link contacts to clients). Any org member can run it.

- **Server** (`src/server/discovery.ts`, `"use server"`):
  - `previewDiscovery({ includeAlreadyScanned?, period? })` — single read-only scan. Pulls eligible rows (`organizationId = activeOrg AND parseStatus = 'complete'` + the `discoveryScannedAt` eligibility filter + an optional period cutoff), aggregates `metadata_json.companies` (any provider) into `ClientCandidate[]`, extracts `{email, name}` participants via `extractParticipants` into `ContactCandidate[]`, infers a same-run `webUrl` for each new company (when a contributing row carries a participant email whose second-level domain normalises to the company key), and builds `LinkProposal[]` matching DB-clients-with-webUrl + new-companies-with-inferred-webUrl against DB-unlinked-contacts + new-contact-candidates. Returns `{ scannedRowCount, scannedRowIds, clientCandidates, contactCandidates, linkProposals, nativeNames, phones }`. **No writes.**
  - **Contact dedup is email-keyed.** A participant whose email already belongs to a contact is dropped (never surfaced, and re-checked at apply). `name` / `name_native` are NOT part of the dedup key — email is authoritative. **Possible-duplicate guard:** a candidate whose email is *not* yet a contact but whose technical OR native name matches an existing contact's technical OR native name **at the same email domain** is flagged with `possibleDuplicate` (the matched contact) and rendered pre-unchecked + amber-badged in `<DiscoverDialog>`, so the operator opts in rather than silently merging. The same-domain corroboration avoids merging distinct people who share a common name (`normalisePersonName` = trim/lowercase/collapse-whitespace, CJK-safe, drops the `(unknown)` placeholder).
  - **Client dedup is normalised-name-keyed, with a same-vs-branch prompt.** `normaliseCompanyName` strips legal suffixes + punctuation, so `IN4COM` / `IN4COM GmbH` / `IN4COM LLC` / `"IN4COM"` all collapse to key `in4com`. A source company whose name is an **exact** (case-insensitive) match to an existing client is silently merged (dropped, as before). A company that matches an existing client **by key but differs in displayed name** (e.g. existing `IN4COM` vs source `IN4COM GmbH`) is kept as a `ClientCandidate` with `possibleDuplicate` set (the matched client) and rendered pre-unchecked + amber-badged — the operator opts in to create it as a **separate client** (e.g. a different country branch). `applyDiscovery` lets a selected `possibleDuplicate` candidate insert even though its key collides with an existing client; non-flagged candidates still block on key collision. Matching is exact-key only (no fuzzy/edit-distance). Trade-off of allowing branches: two clients can then share a normalised key, so same-run link resolution by company key (`newClientKeyToId`) is last-write-wins — an accepted edge case.
  - **Cross-script keying + synonyms (`companyMatchKey` / `personMatchKey` in `src/lib/translit-ru.ts`).** Company dedup keys are now `companyMatchKey` = transliterate-Cyrillic → strip RU legal *prefixes* (ООО/ЗАО/…) → `normaliseCompanyName` (which strips Western *suffixes*). So `АСТ` ≡ `AST` ≡ `ООО АСТ` ≡ `AST LLC` → key `ast`. The new `ClientCandidate.aliases` carries the other spellings discovery folded together (incl. the parser's `organizations[].aliases`), persisted on the created `client.aliases` (`text[]`). Existing-client dedup matches against `client.name` **and every stored alias**. Contact possible-duplicate matching uses `personMatchKey` (translit + token-sort) so `Богданов Евгений` ≡ `Евгений Богданов` ≡ `Bogdanov Evgeniy`. Operators curate `client.aliases` / `contact.aliases` via the edit forms ("Also known as", comma-separated). **`applyDiscovery` also backfills aliases + a blank `webUrl` onto EXISTING clients** (fill-blanks, never overwrite) from two sources: a same-key candidate left unchecked (same company, new spelling), and `DiscoveryPreview.clientEnrichments` — exact-name matches that were never candidates but carry a freshly-discovered website (e.g. an existing `АСТ` with no `webUrl` gains `ast-inter.ru` once a re-parsed email's `organizations[].webUrl` surfaces it). `clientsEnriched` counts both. Runs on every apply incl. "Mark all reviewed".
  - **Company-identity canonicalisation (`src/lib/company-canonical.ts` — pure, unit-tested; consumed by `previewDiscovery`).** Before name-keyed bucketing, a `CompanyUnionFind` collapses spellings that belong to ONE real company so they surface as a SINGLE candidate instead of duplicate clients. Two union signals: (1) **declared aliases** (`organizations[].aliases` — an alias-only mention can't spawn its own candidate even when it also appears in the flat `companies[]` list); (2) **shared website domain** — two names whose parser-attributed `webUrl` (or an existing client's stored `webUrl`) resolve to the same apex domain merge (`www` vs apex normalise via `extractWebsiteDomain`, so `www.ngrsoftlab.ru` ≡ `ngrsoftlab.ru`). Domain edges use ONLY the high-confidence parser/existing webUrl — never the relaxed sole-business-domain inference — so a bad inferred domain can't over-merge distinct companies. Existing clients seed the union-find too, so a new spelling sharing a domain with an existing client canonicalises onto it (→ enriched, never re-created). `applyDiscovery` mirrors the existing-by-domain guard (a selected candidate sharing an apex domain with a live client is not re-created). This fixed the `NGR Softlab` ↔ `NGR Администрация` and `www`/apex duplicate-client cases. `extractCompanySignals` (same module) is the shared `companies[]` + `organizations[]` reader used by both the canonicalisation pre-pass and the main aggregation loop.
  - **Enriched company signal + relaxed webUrl inference.** The email parser emits `metadata_json.organizations: [{name, aliases, webUrl}]` (`ORGANIZATIONS_PROMPT` / `organizationDetailSchema` / `filterOrganizations` in `_shared.ts`) alongside the flat `companies`. Discovery reads both. `webUrl` for a new client resolves in order: (1) parser-attributed `organizations[].webUrl` → high confidence; (2) a participant business-domain whose label matches the company key → high; (3) the **sole** non-freemail domain co-occurring with the company → medium. Step 3 is the relaxed rule that fixes the `АСТ ↔ ast-inter.ru` case the old strict label-match missed (`astinter` ≠ `ast`).
  - **Own-organisation exclusion (`loadOwnOrgIdentity` / `OwnOrgIdentity`).** Discovery loads the CRM owner's OWN identity from the `organization` profile (the `name` / `web_url` / `email` fields the operator filled in on `/account` or `/settings`) and uses it to keep the owner's own company out of attribution. Own **domains** = `extractWebsiteDomain(web_url)` + `extractEmailDomain(email)`; own **company keys** = `companyMatchKey(name)` PLUS, for each own domain, BOTH `companyMatchKey(secondLevelLabel(ownDomain))` AND `companyMatchKey(ownDomain)` (the full domain incl. TLD). All three forms are load-bearing because the display name, the domain label, and the bare domain key differently and the LLM emits any of them as a "company": name `IN4COM` → `in4com`, domain label `in4comgroup` → `in4comgroup`, **full domain `in4comgroup.com` → `in4comgroupcom`** (`companyMatchKey` keeps the TLD). Missing the full-domain key was a real bug: the LLM routinely lists the owner's bare domain in `metadata_json.companies` (e.g. `in4comgroup.com`), which then leaked in as a client candidate, **stole the relaxed sole-business-domain webUrl inference** (claiming an unrelated participant's domain like `ast-inter.ru`), and made the contact↔client domain link **ambiguous** so auto-linking silently failed — see the АСТ ↔ `e.bogdanov@ast-inter.ru` case. Effects: (1) company signals whose key is own are skipped → the owner never becomes a client candidate and never gets enriched, and `rowCompanyKeys` excludes it so contacts in the thread aren't attributed to the owner; (2) participants on an own domain are filtered out at extraction → the owner's mailbox never becomes a contact, never feeds webUrl inference, never anchors a link; (3) webUrl inference (parser URL + sole-business-domain) ignores own domains → an external company in a thread addressed TO the owner no longer inherits the owner's website (the bug where `Moon Realty` got `in4comgroup.com` because the owner's recipient domain was the only non-freemail domain present); (4) `clientByDomain` and `applyDiscovery`'s create filter also drop own domains / own keys (defence against legacy bad data + stale payloads). This is why filling in the org website/email on `/account` materially improves attribution quality.
  - **Linking — domain-first, company-attribution fallback (`LinkProposal.matchedVia`).** (A) the contact's email domain matches a client website domain (existing + new candidates, business-email only) → high (medium if ambiguous). (B) when no domain match, a contact whose scanned rows **consistently** attribute them to exactly one matchable company is linked to that client. Consistency uses an **attributing-rows denominator** — only rows that mention ≥1 matchable (non-own) company count toward the denominator, so a single owner-only email in the thread (which now carries no non-own company key) doesn't veto an otherwise-clear attribution. (B) is what links `e.bogdanov@ast-inter.ru` to client `АСТ` even when the client has no `webUrl`. **Freemail contacts ARE allowed through (B)** — at `low` confidence (pre-unchecked) — because own-org exclusion means the single remaining company in an inbound thread is the external one; this is what links an external `gregnblack@gmail.com` to `Moon Realty` when the gmail sender writes to the owner about one company. Business-email single match → `medium`; ambiguous (>1 client) → `low` + `ambiguous`. One proposal per contact.
  - **Confidence (`DiscoveryConfidence = high|medium|low`) on every candidate + link** gates the review UI: high/medium pre-checked, low pre-unchecked (operator opts in). `<DiscoverDialog>` renders a `ConfidenceBadge` per row; garbage-level signals are dropped before they ever become a candidate (no "none" tier).
  - `applyDiscovery(input)` — sequential apply (Neon HTTP has no transactions). Order is load-bearing: **clients → enrich-existing → contacts → links → stamp**. Inserts selected companies (`status='initial'`, `funnelPhase='awareness'`, `webUrl = inferredWebUrl || null`), inserts selected contacts (`status='initial'`, name from override → displayName → `"(unknown)"`), resolves each selected `(ContactRef, ClientRef)` link against the just-created entities (dropping unresolvable refs + refs to already-linked contacts), then stamps `discoveryScannedAt = now()` on **every** `scannedRowIds` row — including empty-yield rows — so they aren't re-scanned forever. **Revive vs insert**: a selected candidate whose key (client) / email (contact) matches a soft-`deleted` row updates that row in place (`status → initial`, reusing its id) instead of inserting — see § "Schema" above; the result carries `clientsRevived` / `contactsRevived` counts (surfaced in the apply toast). Each step re-checks its dedup key at apply time, so re-running after a partial failure is safe (no duplicates / orphans).
  - **`extractParticipants(meta)`** reads THREE shapes in order, all sharing the same email-keyed dedup: (1) canonical `metadata_json.participants: [{email, name}]` (written by gchat / gdrive sync — see "Per-provider participant extraction" below), (2) the Nylas envelope `from / to / cc / bcc` arrays (fallback for old Nylas rows), (3) `metadata_json.mentionedPeople` — LLM-extracted body mentions written at parse time by every parser (see "LLM-extracted `metadata_json.mentionedPeople`" below). Automated addresses (`isAutomatedEmail`) are dropped; emails dedup lowercased (longest name wins).
- **Schema column** — `source_item.discovery_scanned_at` (nullable timestamp) replaces the two legacy stamps (`client_discovery_scanned_at` / `contact_discovery_scanned_at`), collapsed via `scripts/migrate-discovery-scanned-at.ts` (idempotent, information_schema-guarded: add → backfill → drop). Backfill defaults to `LEAST(old1, old2)` (conservative — NULL when either side was unfinished); pass `--skip-backfill` to leave the new column all-NULL so every row re-surfaces on the first discovery run. Cleared by `reparseSourceItem` so re-parses re-trigger discovery. Default eligibility filter: `discoveryScannedAt IS NULL OR parsedAt > discoveryScannedAt`, bypassed by the dialog's "Re-scan already-reviewed items" checkbox.
- **API**:
  - `POST /api/discovery/preview` — body optional `{ includeAlreadyScanned?, period? }` (tolerates empty / non-JSON body). Returns `DiscoveryPreview`.
  - `POST /api/discovery/apply` — body `ApplyDiscoveryInput` (`selectedClientKeys`, `selectedContactEmails`, `contactNameOverrides`, `selectedLinks`, `scannedRowIds`, `candidates`). Returns `ApplyDiscoveryResult` (`clientsCreated · clientsEnriched · contactsCreated · linksApplied · scannedRowsStamped` + created entity lists). Both route files re-export the relevant types so client components import them without dragging in the `"use server"` module.
- **UI** (`src/components/blocks/discover-dialog.tsx`):
  - `sm:max-w-4xl` dialog. Opening runs the scan; closing without applying stamps nothing.
  - **Period selector** — All time / Last day / Last week / Last month (**default «За день» / `last_day`**, applied to the auto-scan that runs on open + the `reset()`). Toggling re-runs the scan.
  - **Summary bar** — scanned + per-section counts + the "Re-scan already-reviewed items" checkbox (re-runs the scan with the eligibility filter bypassed).
  - **Three collapsible sections** (Companies / Contacts / Links), each with a filter input + "Select/Clear visible" mass-toggle and per-row checkbox (all default checked). Contact rows have an inline-editable name field; Link rows render `contact ↔ client` with a `new` badge for fresh-candidate sides and an amber `ambiguous` badge for multi-match (alphabetical-first client picked).
  - **Footer**: Cancel · Mark all reviewed (apply with empty selections — just stamps) · Apply (N), where N = selected companies + contacts + links.
- **Same-run linking** — because `applyDiscovery` creates clients + contacts before resolving links, a brand-new client (with an inferred `webUrl`) and a brand-new contact discovered in the same scan get linked in one apply step. This is the gap the legacy flow couldn't bridge (fresh clients had no `webUrl`).
- **`initial` status presentation** (clients + contacts): `client-card.tsx` / `contact-card.tsx` render the badge with an orange `STATUS_COLOR.initial` accent; `form-client-edit.tsx` / `form-contact-edit.tsx` expose `initial` as a status option so operators can flip vetted rows to `active`. Manual create still defaults to `active`.
- **Idempotency / re-runs**: each run only scans new + re-parsed rows (unless "Re-scan" is ticked). There is intentionally NO `dismissed_*` table — rejected candidates reappear only if mentioned in a future / re-parsed source item.

### Per-provider participant extraction (`metadata_json.participants`)

What makes contact discovery work across providers (previously Nylas-only). The canonical shape is `metadata_json.participants: [{ email, name }]`, written at sync time:

- **Google Chat** (`src/server/sync/gchat.ts`) — `listChatMembersDetailed(creds, spaceId)` (in `src/lib/google-chat.ts`) fetches space members once per sync (resourceName + displayName + email; the googleapis User type doesn't model `email`, so it's read via a narrow cast). Each message's `sender.name` is resolved against the member map; found → `[{email, name}]`, else `[]` (external / cross-tenant senders have no email). Sender only — silent room members are intentionally not stashed. `listChatMembers` now delegates to `listChatMembersDetailed` and maps down to display names.
- **Google Drive** (`src/server/sync/gdrive.ts`) — `lastModifyingUser` added to the `files.list` fields; participants built by walking `file.owners[]` then `file.lastModifyingUser`, deduped by lowercased email.
- **Nylas** — the envelope `from/to/cc/bcc` arrays are read directly by `extractParticipants` as a fallback, so old rows still work without re-sync.
- **WhatsApp / dropoff / aichat** — no email channel; left as-is (WhatsApp phone-as-contact is a deferred follow-up).
- **Auth**: gchat member listing needs `chat.memberships.readonly` in DWD (already in `APP_SCOPES`); failure is graceful (`[]`, discovery still works for companies).

### LLM-extracted `metadata_json.mentionedPeople` (parse-time)

The third participant source — body-mentioned third parties the parser's existing Gemini call surfaces (no extra LLM round-trip). Closes the gap the sync-time `participants` field misses: people referenced *inside* a body (e.g. "John Donn (john.donn@acme.com) handles that on our side") rather than on the envelope.

- **Shape** — `metadata_json.mentionedPeople: { name, email, organization, phone, confidence }[]` (`phone` quoted from the body when present — business card / contact block — else empty; fed into contact-phone discovery). The `MentionedPerson` type + `mentionedPersonSchema` + `MENTIONED_PEOPLE_PROMPT` + `filterMentionedPeople` all live in `src/server/parsers/_shared.ts`; `MetadataAnalysis` carries the field so `markParsed` / `insertParsedChild` spread it into `metadata_json` automatically (no DB migration — `metadata_json` is `jsonb`).
- **Confidence rules** (in the schema/prompt the model reads): **high** = email quoted in the body OR organization explicitly attributed ("CEO of Acme", "Acme's John"); **medium** = org inferred from the author/sender's own affiliation ("my colleague Jane" said by someone at Acme). Anything fuzzier the model is told to **OMIT**, not emit — without that instruction it dumps every name and pollutes discovery.
- **v1 persist contract** — `filterMentionedPeople` (server-side, post-LLM, pre-persist) keeps only `confidence === "high"` entries with a non-empty email, deduped by lowercased email. Medium-confidence + email-less entries are emitted by the model (future use) but stripped before write, because discovery dedups by email and the contact table is email-keyed. Deferred surfaces tracked in PHASE2.md #14.
- **Per-parser prompt addenda (required)** — every parser appends `MENTIONED_PEOPLE_PROMPT` to its `system` string PLUS a short provider-specific addendum naming who the author/sender is for that provider (From: address for email, `msg.sender` for chat, speakers for audio/video, extracted `senders` for pdf/image/office, file owner for drive, conversation participants for whatsapp). Without it the model double-counts the author against the envelope pass or can't infer the medium-confidence org. The 9 parsers wired: text, chat, audio, video, pdf, image, office, drive, whatsapp. `dropoff.ts` is a pure dispatcher and inherits from the format parsers. Drive composes the addendum once inside `runMetadataExtraction` (shared by two callers); video's two analysis blocks share ONE `mentionedPeople` list via the `sharedFields` spread.
- **No backfill** — new parses get the field; old rows surface mentions only on re-parse or via the "Re-scan already-reviewed items" toggle in `<DiscoverDialog>`. Same posture as the sync-time `participants` rollout.

### Native contact names + phones (`contact.name_native`, `contact.phone`)

Email envelopes carry a **technical** participant name (admin-defined, often English/romanized — e.g. `"Mueller, Thomas"`), while the person's **real** name + phone appear in the body signature (`"Thomas Müller"`, `"张伟"`, `+49 30 1234-567`). We recover both and attribute them to the contact by email: `contact.name` = technical (stable identity, what address books match), `contact.name_native` (nullable) = local-language form, `contact.phone` = phone from the body.

- **Two extraction sources, both email-keyed:**
  1. **Envelope participants (`text.ts`, email-only)** — `participantDetails: { email, nativeName, phone }[]` on the email parser's analysis schema. The LLM gets the From/To/Cc/Bcc addresses in the prompt and pairs each to the real name + phone it finds in the body (signature / contact block), original script, no romanizing. `filterParticipantDetails` (in `_shared.ts`) keeps only entries whose email is an actual envelope address carrying at least one usable detail (deduped: longest native name, first plausible phone). This is the path for the **sender's own** name/phone (the `mentionedPeople` prompt excludes the author).
  2. **Body-mentioned third parties (all 9 parsers)** — `phone` is now a field on `mentionedPersonSchema`, so any high-confidence + email-bearing mention (business card in an image, "call John at +49… (john@acme.com)" in any body) carries a phone for free.
  - Both persist into `metadata_json` (`participantDetails` / `mentionedPeople`) → **searchable** via `searchSourceItems` + `listClientContent`. `participantDetails` is **optional** on `MetadataAnalysis` (only `text.ts` populates it); `phone` rides on the shared `mentionedPersonSchema` so all parsers emit it. Phones go through `cleanPhone` (trim, strip non-dialable chars, require ≥ 7 digits — no E.164 normalisation; the as-written local format is preserved).
- **Attribution** is the LLM's job (which of from/cc/bcc signed off? whose card is this?) — it sees envelope + body, so it pairs name/phone → email; regex can't. Confidence-gated: omit when unsure.
- **Discovery** — `previewDiscovery` accumulates email→nativeName and email→phone maps across scanned rows (kept for *all* emails, including ones already contacts; phone sourced from both `participantDetails` and `mentionedPeople`). The preview returns the full `nativeNames[]` + `phones[]`. `applyDiscovery` sets `name_native` + `phone` on newly-created contacts AND **backfills blanks on pre-existing contacts** (the sender is usually already a contact) — fill-blanks-only, never overwrites a human-edited value. `ApplyDiscoveryResult.{nativeNamesEnriched, phonesEnriched}` count the existing-contact fills.
- **UI** — `form-contact-edit.tsx` has "Native name" + "Phone" fields; `contact-card.tsx` shows native name as a subtitle + phone in the body; the Contacts-tab name filter matches `name` OR `name_native`. (Phone needed no new UI — the column + form field already existed.)
- **DB / backfill** — `contact.name_native text` added via `scripts/add-contact-name-native.ts`; `contact.phone` already existed. No data backfill — values populate on new parses / re-parse / "Re-scan already-reviewed items", consistent with the participants + mentionedPeople rollout.

### Web lookup (Gemini + grounded `google_search`)

Manual button on each client card (`<Globe>` icon next to the edit pencil) and on the client detail page (full "Lookup on web" button next to "Edit"). Runs a two-pass Gemini call to find the company on the web and proposes updates to its contact fields. Any org member can use it.

- **Server** (`src/server/clients.ts → lookupClientOnWeb(clientId)`):
  - Loads the target client + its active contacts (names, emails, positions). Builds a multi-line "what we know" block from every non-empty client/contact field — gives the model maximum disambiguation context.
  - **Pass 1 — research**: `generateText` with the `google_search` tool (`google.tools.googleSearch({})`) via the AI Gateway. Model writes 2-4 paragraphs of freeform research notes. `result.sources` (URL + title array) is captured and threaded through to the UI for transparency. `stopWhen: stepCountIs(5)` bounds the search loop.
  - **Pass 2 — structured extraction**: `generateText` with `Output.object({ schema })` (no tools), feeding it the research notes + the client's current fields. Returns up to 3 candidate matches. Schema fields per candidate: `name`, `email`, `phone`, `address`, `webUrl` (empty string when not found in research), `confidence` (`high` | `medium` | `low` self-rating), `whyMatch` (one-line disambiguation hint).
  - **Why two passes**: combining `tools` with `Output.object` is brittle across providers / SDK versions. Splitting into two known-good calls trades ~2× cost + latency for reliability. Acceptable for a manual button.
  - **No DB writes.** Caller submits the user's chosen + edited fields via the existing `PUT /api/clients`.
- **API**: `POST /api/clients/[id]/lookup` — `maxDuration = 60` (grounded search has occasional long tails). Returns `{ candidates, sources, notes }`. 403 / 404 / 400 error codes mapped from the same error messages the rest of `clients.ts` throws.
- **UI** (`src/components/blocks/client-lookup-dialog.tsx`):
  - Two-step modal that progressively narrows. Phases: `searching` → either `select-candidate` (when N > 1) → `edit` → `saving`, OR `searching` → `edit` directly (when N = 1, skipping the selector unconditionally).
  - **Searching** — spinner + "Searching the web for {name}…". Single fetch.
  - **Candidate selector** (only shown for ≥ 2 candidates) — vertical stack of clickable cards: bold name, address line, clickable website link (opens in new tab without selecting), confidence chip (green/amber/red), and the model's one-line "why this match" hint. "None of these" closes the modal without writes.
  - **Edit form** — shadcn-Input row per writable field (`name` / `email` / `phone` / `address` / `webUrl`). Pre-filled with `candidate[field] || currentClient[field]` so an empty search result never blanks an existing value. Each field gets a small **changed** badge when its draft value differs from the current client value. A low-confidence candidate triggers a top warning banner. "Back" returns to the selector (only when there were multiple candidates).
  - **Sources block** — bottom of the modal, dedupes URLs by host (so a 5-link LinkedIn doesn't dominate), shows up to 4 with a "+N more" expander. Every source link opens in a new tab.
  - **No-results state** — distinct error view ("No matches found — try refining the client name first") with a Close button. Doesn't blank the modal.
  - **Save** — PUTs `/api/clients` with `{ id, name, email, phone, address, webUrl }`. Empty strings convert to `null` so the column ends up cleared rather than holding `""`. Toasts success and calls `onSaved()` so the parent (card grid or detail shell) re-fetches.
- **Out of scope (for v1)**: doesn't touch `funnelPhase`, `status`, or `userId`. Doesn't create/update contacts (that's a separate next-step feature). No rate limiting per client per day — same posture as the existing manual-trigger AI buttons (Discover, Parse). When abuse / cost becomes a real concern, add a per-client cooldown column.

### Batch web enrichment ("Обогатить клиентов")

Batch driver over `lookupClientOnWeb` that fills the missing contact fields (website / email / phone / address) of every not-yet-processed client. Toolbar button on `/clients` beside "Найти в источниках". Spec: `refs/enrich-clients.md`.

- **Schema** (`client`): two nullable columns — `enrichment_status` enum (`enriched` | `review` | `no_match`) + `enrichment_candidates jsonb` (`ClientLookupCandidateJson[]`). **`NULL` is the resumability anchor**: NULL = never processed OR last run failed → the worklist; a thrown error during processing leaves the row NULL so the next run retries only it (no separate "failed" state). `review` parks the candidates so the manual dialog needs no new web call. Applied additively (enum + columns + `client_enrichmentStatus_idx`) — no `drizzle-kit push` (it would drop the Search V2 DDL columns).
- **Server** (`src/server/clients.ts`): `listPendingEnrichIds(limit=200)` → `{ ids, total }` (worklist = org + `enrichment_status IS NULL` + `status IN ('active','initial')` + ≥1 blank fillable field, ordered by `updated_at`, capped 500). `enrichClientFromWeb(clientId)` processes ONE client: raced-to-complete → `skipped`; 0 candidates → `no_match`; **exactly one `high`-confidence candidate → auto-apply** (fill-blanks-only) → `enriched`; otherwise → `review` (parks candidates). **Does NOT catch the lookup error** — it must propagate so the row stays NULL and retries. `listEnrichReview()` / `countEnrichReview()` drive the dialog + badge. `resolveEnrichment(clientId, {candidateIndex}|{skip})` applies a human's pick (fill-blanks-only) or dismisses (`no_match`). The **fill-blanks-only `candidatePatch`** is the load-bearing invariant: only currently-empty fields are filled, never overwriting user-entered data → idempotent + safe to re-run.
- **API**: `GET /api/clients/enrich/pending?limit=` → `{ ids, total }`; `GET /api/clients/enrich/review` → `{ rows }`; `POST /api/clients/[id]/enrich` (`maxDuration=60`, one client/request) → `{ outcome, filledFields, candidateCount }`; `POST /api/clients/[id]/enrich/resolve` (`{ candidateIndex } | { skip:true }`). Static `enrich/*` segments resolve before the dynamic `[id]` slug (ids are UUIDs, never literally "enrich").
- **UI** (`src/components/blocks/client-enrich-control.tsx`): browser-driven loop — `GET pending` then `POST …/enrich` **one client at a time** with a 4s inter-client gap (free-tier pacing; **this is the only mode — no speed options**) + an `AbortSignal.timeout(75s)` backstop. Each client commits server-side as it finishes, so cancel / closed tab loses nothing already done (re-run continues the remainder). Counts poll on mount + every 30s (paused while running) + on `refreshKey`. While running, the primary button swaps for an inline progress chip (spinner + `done/total` + Отменить). "На проверке (K)" opens the **review dialog** (Phase B, manual disambiguation): per parked client, its name + a vertical stack of candidate cards (reusing the per-client lookup styling — confidence chip + address + email/phone + website + `whyMatch`) + a "Пропустить" button. Picking a card → `resolve {candidateIndex}`; Пропустить → `resolve {skip:true}`. **No new web calls** — replays the parked candidates.

### Client content match rule

The `/clients/[id]` "Client Content" table uses a deliberately broad relevance heuristic. It is implemented in `src/server/client-content.ts` (`listClientContent`) and surfaced via `GET /api/clients/[id]/content`. Fix the rule there if business needs change.

- **Hard pre-filter** (always applied): `parseStatus = 'complete' AND r2UploadStatus = 'complete'`. Rows still in flight or skipped are invisible. The user explicitly chose to scope this view to R2-backed rows only.
- **Match terms collected** (every non-empty client/contact field becomes one ILIKE term, lowercased):
  - `client.name`
  - `client.address`
  - `client.webUrl` — normalised to host-only via `normalizeWebsite()`: strip `^[a-z]+://`, strip `^www\.`, drop path/query/fragment, lowercase. So `https://www.acme.com/about` becomes `acme.com`.
  - `contact.name` (every contact attached to the client, **including suspended** — keeps history queryable after a contact is archived)
  - `contact.email`
- **Min term length is 2.** Single-character names won't degrade the result with millions of false hits.
- **Match clause**: each term is OR'd into a single relevance predicate of the form `metadata_json::text ILIKE '%term%'`. The full predicate is `(t1 OR t2 OR …)`. A row is included if **any** term matches **any** part of `metadata_json::text`. We do NOT yet probe specific JSON fields (companies / mentions / urls / senders / recipients) — the cast-to-text + ILIKE approach catches all of them in one cheap pass and avoids the cost of per-field expression indexes. Trade-off: a contact email like "ceo@a.co" can collide with unrelated text. Acceptable for v1; can be tightened to per-field probes later if precision becomes an issue.
- **Body markdown is NOT searched** — only `metadata_json`. Full-text search over the parsed bodies will land later (probably as a Postgres tsvector column or an external index).
- **Filter composition**: the relevance clause is AND'd with the route-level filters (sourceId, mime bucket, q, date range). `q` is a refinement, never an expansion — it narrows from the relevance-matched set.
- **Empty-signals fallback**: if every match term is empty (no name, no address, no website, no contacts), the listing returns `{ rows: [], total: 0, matchTerms: [] }` and the table shows a "Add an email, website, or contact to start matching content." empty state.

## Products

Org-scoped product catalog. First tenant (IN4COM, org `1dNkC4rBtl9FEvnK95Svlfs63oSU57Ni`) is a wine/spirits catalog (~16.8k rows) loaded from a 42-column spreadsheet. **Read-only for now** — a paginated, server-searched table; management functions (create / edit / stock adjust) are a deliberate later step.

### Schema (`src/db/schema.ts` → `product`)

Spreadsheet columns map into three buckets (row 1 = DB names, row 2 = UI labels, row 3+ = data):
- **Main attributes** → real columns: `name` (C, notNull), `category` (D), `webPageUrl` (E), `price` (F, `numeric(14,2)` — drizzle returns it as a string; API/UI parse to number), `imageUrl` (H). `total_stock` (AA) is a real `integer` column so the table sorts/shows it cheaply.
- **`accountingMetadata`** jsonb (cols A-B) → `{ code, barCode }` — accounting-system IDs kept opaque so that field set can evolve without a migration. (Barcodes arrive as `"'3258436200007'"` in the export; the loader strips the leading quote.)
- **`additionalMetadata`** jsonb (cols I-Z + the dropped `price_range` col G) → catalog attributes keyed by DB name (description, color, country, region, taste, …). `price_range` was explicitly dropped as a column but folded here so no data is lost.
- **`stockMetadata`** jsonb (cols AB-AP) → per-location breakdown as `[{ key, label, count }]` (one entry per warehouse/store, keeping each location's display name).
- `status` reuses the shared `entity_status` enum (default `active`); org-scoped via `organizationId`. Indexes on org / status / name / category.

### Server (`src/server/products.ts`)

- `listProducts({ q?, limit?, offset? })` — org-scoped (`requireOrgContext`), `status = 'active'` only, ordered by name. Returns `{ rows: ProductRow[], total }`. `limit` clamped to 1–100 (default 25), `offset ≥ 0`. Both the page and count run in one `Promise.all`. **This is the only product surface today — no mutations yet.**
- **Filter attributes — single vs multi-value.** Most attribute dropdowns (`type`/`sugar`/`year`/`aging`/`bottleVolume`/`appelacion`/`rating`) are single-value (`= ?`). The three **export-relevant** attributes are **multi-select** and take ARRAYS — `countryNames?: string[]` + `regions?: string[]` + `colors?: string[]` — each gated via `hardAttrMulti` as a `col IN (…)` clause (single + array forms merged & deduped; `region` is multi-only). `region` has no single-value param. The single `color`/`countryName` params still exist for back-compat but the page sends only the arrays.
- `listProductFilterOptions()` returns the distinct dropdown values per attribute PLUS, for the multi-selects, **`region: string[]`** (all regions) and **`regionsByCountry: Record<string,string[]>`** (built from distinct `(country_name, region)` pairs) so the UI can scope the region choices to the currently-selected countries. Only non-empty values are returned (blank-valued rows are NOT represented by any option — see the "all-selected ⇒ omit" rule in the UI section).
- **Search (Search V2 — hybrid, cross-script)** — the search box `q` and the order-from-request wizard's `terms[]` feed ONE ranked path. The query is normalized in SQL — Cyrillic→Latin transliteration (`translit_cyr_lat`) + `immutable_unaccent` — then **brand-alias-expanded** from the `product_alias` table, then two retrievers run over the gated set and are fused by **Reciprocal Rank Fusion** (`rrf = Σ 1/(60+rank)`): (a) **FTS-OR** over the weighted `search_vector` tsvector via `to_or_tsquery()` (OR semantics — one missing normalized token can't blank the match), (b) **trigram** `word_similarity(qexp, name_norm) ≥ 0.3`. Deterministic tiebreak `char_length(name), name, id`. **Why:** the catalog is ~90% Latin names but ~95% of queries are Cyrillic, so the old pg_trgm fuzzy layer scored 0 cross-script — V2 moves both sides into one script so грей гуз→Grey Goose, вионье→Viognier, опкхард→Oakheart all resolve at rank #1 (eval: Recall@1 22%→98%). **Gates** (hard AND): org, `status='active'`, junk-row exclusion, `is_drink` (merch hidden unless `includeMerch`), price, in-stock, explicit `volumeMl`, plus the manual filter dropdowns (category/type/color/… — an explicit operator choice still narrows). **Soft hints** (boosts in ORDER BY, never gate): `catHint`/`colorHint`/`sugarHint`/`countryHint`/`regionHint`/`wantGift` — additive boosts a query can carry. **Currently nothing sends them** — the order-from-request wizard was switched to pure text + price (the catalog's category/type/color are mis-attributed; see § "Order from request"), so these params are retained only for a possible future discovery surface. The manual dropdowns remain HARD filters. A query that matches nothing falls back to the gated catalog (never-blank). Empty `q`/`terms` → plain filtered listing ordered by name. The DDL (generated columns + functions + indexes), alias seed, and eval harness live in `scripts/search-v2/` + `scripts/eval/`; the canonical spec is `refs/search-v2-plan.md`. Pre-V2 engine recorded in `refs/search_spec.md`.

### API (`src/app/api/products/route.ts`)

- `GET /api/products?q=&limit=&offset=` → `{ rows, total }`. Re-exports `ProductRow` / `ProductStockLocation` / `ListProductsResult` / `ProductFilterOptions` so the client imports them without dragging in the `"use server"` module. 403 on unauthorized / no active org. No POST/PUT/DELETE yet.
- `?categories=1` → `{ categories }`; `?filterOptions=1` → `{ options }` (incl. `region` + `regionsByCountry`).
- **Multi-value filters are REPEATED params, read via `getAll`** — `?colors=A&colors=B`, `?regions=…`, `?countryNames=…` (the listing) — **never comma-joined**: some region values legitimately contain commas (e.g. `"АЛЬГЕРО,САРДИНИЯ"`), so a delimiter would split one value into two non-matching tokens.

### Price-list export (`src/server/products-export.ts` + `src/app/api/products/export/route.ts`)

- `GET /api/products/export?countries=&regions=&colors=` → an **XLSX workbook, one sheet per country**, each a hierarchical price list grouped left→right **region → color → vendor → name** (repeated group cells blanked, outline-style). Products with no `country_name` land in a **«Без страны»** sheet (always last). Built with **ExcelJS** via `buildPriceListWorkbook(filters?)`; `maxDuration = 120`. Mirrors `refs/price-list.xlsx`. CSV is intentionally not offered (the format is inherently multi-tab).
- **Only country / region / color filter the export** — they ARE the price-list hierarchy. Every OTHER active page filter (type/sugar/year/price/in-stock/…) is deliberately ignored by the export. No country/region/color set → the whole active catalog. Like the listing, the route reads the three params via `getAll` (repeated, comma-safe).

### UI (`src/app/(protected)/products/page.tsx`)

- Client component, **server-driven pagination** (25/page) — never pulls the whole catalog. Debounced search box (300ms) sets the server `q` and resets to page 1; an out-of-order-response guard (`reqIdRef`) keeps the latest fetch authoritative. Prev/next pager with "Page X of Y" + a "showing N–M of T" counter.
- Plain shadcn `Table` (no cards): **Image** (remote `<img>` thumbnail from the ast.wine CDN — plain `<img>` avoids configuring next/image for an arbitrary remote host; `ImageOff` placeholder when null), **Name**, **Category**, **Page** (external-link icon → `webPageUrl`), **Price** (right-aligned, `ru-RU` grouping), **Stock** (`totalStock`). `price_range` is intentionally not shown.
- **Filter row.** Single-value attributes use `FilterSelect` (shadcn `Select`, sentinel `__all__`). The three **export-relevant** attributes — **Страна / Регион / Цвет** — are **multi-selects** (`MultiFilterSelect`) and sit FIRST with **yellow labels** (`accent`) to mark them as the only filters the price-list export honors. **`MultiFilterSelect` is built on Popover + plain `<Checkbox>` rows, deliberately NOT cmdk's `<Command>`** — cmdk tracks a single highlighted item and re-runs its own filter on every state change, which dropped picks in a multi-select (only the first click per open registered). Each checkbox toggles independently via a **functional `setFilters` updater** (`toggleMulti`/`toggleCountry`) so no click can race on a stale snapshot. **Region is the child of country**: its options (`availableRegions`) are scoped to the selected countries (`regionsByCountry`); pruning of orphaned regions runs ONLY when a country is **removed** (adding a country can only widen the set, so it must never drop a pick).
- **"All-of-an-attribute selected ⇒ omit that filter" (`appendUnlessAll`).** A `col IN (…)` gate can't match a NULL/empty value, so selecting every color would silently drop products with no color. When a multi-select's selection covers all its options the param is omitted entirely (no constraint), which keeps the blank-valued rows. Applied to color, region, and country in BOTH the listing fetch and the export. (Partial selection still narrows normally.)
- **Export button** ("Экспорт прайс-листа", top-right of the catalog card) forwards only the current country/region/color selections (via `appendUnlessAll`) to `/api/products/export` and downloads the XLSX (fetch-to-blob + spinner + error toast).
- Sidebar: "Products" entry (`Package` icon) sits between Clients and Tasks in `src/components/blocks/app-sidebar.tsx`.

### Data migration (`data_migration/`)

- `load_products.py` — one-shot loader. Reads `refs/product-ai-collection-catalog 2.xlsx` via openpyxl, maps the 42 columns per the bucket scheme above, inserts via psycopg (reads `DATABASE_URL` from `../.env`). ## Orders

Org-scoped sales orders. Created by an org user, assigned to exactly one client, holding a list of product lines drawn from the catalog. Surfaced as a second **Orders** tab on `/products` (next to Product Catalog). Today the tab is **read-only listing** (the same server-paginated/searched table shape as the catalog); create/edit/line-item UI is a deliberate later step, but the full create/update/status server layer already exists.

### Schema (`src/db/schema.ts`)

- **`order_status`** pgEnum: `draft` → `awaiting_client` → `confirmed` → `finalized` / `cancelled`. The `awaiting_client` ↔ guest-confirmation-link ownership model is specced in `refs/spec-guest-order-link.md`; the link layer itself isn't built yet, but the statuses are in place for it.
- **`order`** — `order_date` (timestamp, business date, distinct from `created_at`, default now), `description` (nullable), `status` (default `draft`), `total_amount` (`numeric(14,2)`, denormalised Σ line `position_price`, recomputed on every item change — the **pre-discount** catalog subtotal), **`discount_percent`** (`numeric(5,2)` NOT NULL default 0 — the per-client discount snapshotted onto the order; see § "Order discount"), `currency` (text, default **`RUB`**, normalised to a 3-letter ISO code server-side), `client_id` (notNull, FK `restrict` — clients are soft-deleted, never hard-deleted), `user_id` (creator, cascade), `organization_id` (cascade). Indexes on org / user / client / status / order_date. `discount_percent` added via `scripts/add-order-discount.ts` (additive, NOT `drizzle-kit push`).
- **`order_item`** — one product line: `order_id` (FK `cascade` — deleting an order clears its lines), `product_id` (FK `restrict`), `quantity` (int, default 1), `unit_price` + `position_price` (both `numeric(14,2)`). `unit_price` is **snapshotted** at add-time (catalog price drifts); `position_price = unit_price × quantity` is stored so reads never recompute.

### Server (`src/server/orders.ts`)

All org-scoped via `requireOrgContext()`; per-row mutations assert tenant scope via `assertOrderInOrg` / `assertClientInOrg`. `numeric` columns are written as fixed 2-decimal strings (`money()`); currency via `normalizeCurrency()` (defaults `RUB`, throws on non-ISO).
- `listOrders({ q?, status?, limit?, offset? })` → `{ rows, total }`. Joins `client` for the name; orders by `order_date` desc. Search is a single ILIKE OR'd across `order.description` + `client.name` (kept simple — richer filters later). `limit` clamped 1–100 (default 25).
- `getOrder(id)` → `OrderDetail` (header + line items with product names joined + `clientEmail` + the most-relevant guest-link `link` meta) or null (route → 404).
- `createOrder({ clientId, description?, orderDate?, currency?, items? })` — always creates a **draft**. `resolveItems()` validates every `productId` against the org's catalog, fills `unitPrice` from the product's current price when omitted, computes each `positionPrice` + the order total, then inserts the `order` + `order_item` rows.
- `updateOrder(id, …)` — **draft-only** content update (throws unless the order is a draft — FR-10). When `items` is provided it **replaces the whole line set** (delete + re-insert) and recomputes the total. Status transitions live in `@/server/order-links`, not here.
- `listOrderClientOptions()` — `active` + `initial` clients (with `email`, to prefill the send-link recipient, **and `discount`** so the builder previews the discount live) for the builder's client picker.

### Order discount (per-client %)

A per-client discount applied to the order's catalog subtotal. The discount lives on the **client** (`client.custom_fields.discount`, a whole % 0–100, all orgs — see § "Clients & Contacts"); each order **snapshots** it.

- **Storage = percent only; amounts derived.** `order.discount_percent` is snapshotted from the client; `total_amount` stays the pre-discount subtotal. The **discount amount** + **discounted total** are DERIVED on read by `computeOrderDiscount(total, percent)` (in `src/lib/orders-format.ts`, pure + client-safe) — never stored. So a guest qty change (which recomputes `total_amount` via `recomputeOrderTotal`, leaving `discount_percent` untouched) re-applies the discount at the same percent with no extra write.
- **Re-sync on every internal save** (decision, not frozen): `createOrder` + `updateOrder` re-read the (possibly newly-assigned) client's current discount via `clientDiscountPercent(clientId)` and write `discount_percent`. So fixing a client's discount and re-saving the order applies it; past orders left untouched keep their snapshot. Guest qty edits never change the percent.
- **Surfaced** on: `OrderRow` / `OrderDetail` / `GuestOrderView` (all carry `discountPercent` + derived `discountAmount` + `discountedTotal`); the orders table (**Итого / Скидка / Итого со скидкой** columns); the order builder panel (live preview — derives from the selected client option's discount, falling back to the order's stored percent until options load); and the guest order page. The guest derivation reuses the same `computeOrderDiscount` so the three surfaces never drift.

### API (`src/app/api/orders/route.ts`)

- `GET /api/orders?q=&status=&limit=&offset=` → `{ rows, total }`. `?id=…` → `{ order }` (404 if missing). `?clientOptions=1` → `{ clients }`. Re-exports `OrderRow` / `OrderItemRow` / `OrderDetail` / `OrderClientOption` / `ListOrdersResult`.
- `POST /api/orders` — create a draft (required: `clientId`). `PUT /api/orders` — draft-only content update. Status transitions are on `POST /api/orders/[id]/link` (see § "Guest order-confirmation links") so the link lifecycle stays in one place.

### UI

- **Shared display helpers** — `src/lib/orders-format.ts` exports `ORDER_STATUS_LABEL` / `ORDER_STATUS_COLOR` / `ORDER_STATUSES` / `isOrderEditable(status)` / `isValidEmail` / `formatOrderAmount` / `formatOrderDate`, used by the table, the builder, and the guest page so status styling + currency formatting never drift. `isOrderEditable` is **`draft` OR `confirmed`** — `draft` (not yet sent) and `confirmed` (client returned it, internal user owns it again and may adjust before finalizing); `awaiting_client` belongs to the client, `finalized`/`cancelled` are terminal.
- **Orders list** (`src/components/blocks/orders-table.tsx`) — self-contained client component under the Orders tab. Same server-driven pagination (10/page default) + debounced (300ms) search + out-of-order-response guard (`reqIdRef`) as the catalog table. - **Order builder** (`src/components/blocks/order-builder.tsx`) — the manual create/edit/preview surface. Three exports:
  - `useOrderBuilder({ onSaved })` — page-level hook owning the whole editing session (open/edit state, the client-side `DraftLine[]`, persistence).   - `OrderBuilderPanel` — the **sticky panel** (`sticky top-2 z-20`) rendered atop the Catalog tab while a session is active, so it stays visible while the user searches the catalog. Shows client `Select` + description, an **"Allow only products in stock"** checkbox, the editable line-item table (unit price + qty inline-editable, position + total derived live), the running total, and status-aware actions: Draft → *Save draft · Send to client · Discard*; existing Draft → adds *Cancel order*; `awaiting_client` → read-only + *Re-send link · Back to draft · Cancel*; **`confirmed` → editable** (the client returned it) + *Cancel order · Reopen to client · Save changes · Finalize the order*; `finalized` → read-only + *Reopen to client · Close*; `cancelled` → read-only + *Reopen as draft*. "Send to client" requires a client + ≥1 line.
  - **Stock guard (`stockOnly`)** — a per-session checkbox on the panel. When on, `addProduct` blocks products with null/≤0 catalog stock and any add that would push a line's total over available `totalStock` (toast on block), `setQuantity` caps inline qty edits at stock, and the catalog's per-row Add button is `disabled` for out-of-stock rows. Stock travels on `DraftLine.stock` / `AddableProduct.stock`; `getOrder` now returns `OrderItemRow.productStock` so the guard works on edited orders too. `null` stock = unknown → unconstrained. The order-from-request wizard adds through the same `AddToOrderButton`, so it's covered automatically. UI-only — not persisted (the catalog `totalStock` is the source of truth at add time; no stock reservation).
  - `AddToOrderButton` — per-catalog-row popover (quantity field, default 1) that feeds the product into the active builder.
- **Page wiring** (`src/app/(protected)/products/page.tsx`) — a **"New order"** button sits beside the `TabsList` (visible on both tabs); the `Tabs` is controlled so New Order / Open-from-list switch to the Catalog tab (where the picker lives). The builder state lives at the page level (survives tab switches). The catalog table grows one conditional **"Order"** column (`showAddCol = builder.isActive && !builder.readOnly`) rendering `AddToOrderButton` per row; `catalogColSpan` adjusts the loading/empty spans. On save the page jumps to the Orders tab and bumps `refreshKey`.
- **Ownership model (guest link landed)** — `isOrderEditable` is **`draft` OR `confirmed`**. Once an order is sent (`awaiting_client`) the CLIENT owns it: the builder renders read-only with a link-metadata strip + link actions (Re-send / Pull back to draft / Cancel). "Pull back to draft" revokes the live link and unlocks editing. **`confirmed` (client returned it) is editable again** — the internal user can adjust line items / qty / price before finalizing: *Save changes* (`builder.saveChanges` → persist + re-hydrate, stays open), *Reopen to client* (persists edits first, then re-sends), *Finalize the order* (persists edits first, then confirmed→finalized — internal-only, no-op beyond the status change for now), *Cancel order*. `finalized` → read-only + "Reopen to client"; `cancelled` → "Reopen as draft". Server matches: `updateOrder` allows `draft` **or** `confirmed` (throws otherwise); `PUT /api/orders` stays content-only (status transitions go through `/link`). See the **Guest order-confirmation links** section below.

### Guest order-confirmation links

Capability-based, no-auth client review/confirm loop implementing `refs/spec-guest-order-link.md`. The bearer of an opaque high-entropy token can review + adjust + confirm exactly one order while it's `awaiting_client`.

- **Schema** (`src/db/schema.ts`) — `order_access_link` + `order_link_status` enum (`active · used · revoked · expired`). Columns per spec: `token_hash` (sha256 hex, unique-indexed — raw token never stored), `recipient_email`, `status`, `expires_at`, `last_accessed_at`, `confirmed_at`. A **partial unique index** `(order_id) WHERE status='active'` enforces invariant #1 (≤1 live grant per order); `order` cascade-deletes its grants.
- **Token** (`src/lib/order-link-token.ts`, `server-only`) — `generateRawToken()` (32-byte CSPRNG base64url), `hashToken()` (sha256 hex — fast hash is correct for high-entropy tokens, FR-1/§7), `buildOrderLinkUrl()` (`NEXT_PUBLIC_APP_URL/o/<raw>`).
- **Link layer** (`src/server/order-links.ts`, `server-only`) — the **only** place that mints/revokes grants (invariant #7):
  - `capabilitiesFor(grantStatus, orderStatus)` — pure; `editQuantity = removeItem = confirm = (grant active && order awaiting_client)`. `addItem` never exists (FR-4).
  - `resolveOrderLink(token)` — single public chokepoint (FR-3): hash → grant by `token_hash`, lazily flip `active→expired`, reject `not_found/revoked/expired`, load the order **by `grant.order_id`** (never client-supplied). `used` grants resolve to a read-only confirmed view.
  - Guest mutations `guestSetQuantity` / `guestRemoveItem` / `guestConfirmOrder` — re-resolve + re-check capability every call; writes pin **both** the line id **and** the token-derived `order_id` (IDOR guard, invariant #4); recompute the order total in one SQL statement; bump `last_accessed_at`. Confirm uses an optimistic `UPDATE … WHERE status='awaiting_client'` (Neon HTTP has no transactions — the order update is the source of truth; zero rows ⇒ conflict, FR-6).
  - Internal transitions (org-scoped via session): `sendOrderToClient` (draft→awaiting_client + mint), `resendOrderLink` (rotate), `reopenOrderToClient` (confirmed/finalized/cancelled→awaiting_client + fresh link), `returnOrderToDraft` (pull back + revoke), `cancelOrderAndRevoke` (cancel cascades revoke), `finalizeOrder` (confirmed→finalized, internal-only, no grant change — accounting hand-off is a later step). Sending requires a structurally-valid recipient email (`isValidEmail`), defaulting to the order's client email — `email_required` / `invalid_email` / `conflict` reasons surface to the UI. `ORDER_LINK_EXPIRY_DAYS = 14` (spec §10 default). `getOrderLinkMeta(orderId)` feeds the internal builder's link strip (no raw token — that's shown once at mint).
- **API** — `POST /api/orders/[id]/link` `{ action: send|resend|reopen|pullback|cancel|finalize, recipientEmail? }`; reason→HTTP map (422 email, 409 conflict). The generic `PUT /api/orders` no longer changes status (transitions all go through `/link`).
- **Guest route** (`src/app/o/[token]/`, public — no middleware needed since auth is layout-gated, not middleware) — `page.tsx` RSC (`force-dynamic`) resolves the token → `<GuestOrderReview>` or `<LinkUnavailable>`; `actions.ts` server actions wrap the guest mutations + `revalidatePath`; `guest-order-review.tsx` renders editable qty steppers (key-remounted per commit) / remove / confirm, or a read-only/confirmed view derived purely from `can`.
- **Internal builder** (`order-builder.tsx`) — `SendLinkDialog` collects the recipient email (prefilled from the client), mints, and shows the copyable link **once** (raw token isn't re-derivable). `LinkMetaStrip` shows link status/expiry/last-opened for `awaiting_client`.
- **Rate limiting (FR-11)** — deferred (no Upstash in the project; tokens are 256-bit). A seam to add `@upstash/ratelimit` on `/o/*` later.

### Order from request (LLM-assisted assembly)

An intelligent assistant that turns an unstructured, free-text client request (the kind in `refs/act_client_requests.md` — informal Russian WhatsApp/email messages) into a draft order by splitting it into intent items and walking the rep through assembling each one against the catalog. **No Vercel Workflow** — deliberately modelled as durable rows + a UI wizard over the existing manual order builder (the draft order IS the durable state; a human-paced wizard is the wrong shape for a suspended workflow). The split itself mirrors `deals-discovery` (one `generateText` + `Output.object` call via the AI Gateway), not the orchestration layer.

- **Schema** (`src/db/schema.ts`):
  - `order_request` — durable header for one pasted message: `raw_text`, `comment` (becomes the order description), `status` (`order_request_status` enum: `parsing → ready → assembling → done | abandoned`), `parse_error` (set when the split fails — non-fatal), `client_id` (FK `restrict`, like `order`), `order_id` (FK `set null` — the draft being assembled, null until minted), `user_id`, `organization_id`. The draft `order` owns the real order lifecycle; this status only tracks the assistant session.
  - `order_request_item` — one parsed intent = one wizard step (maps to 0..N order lines): `ordinal` (0-based step order), `raw_snippet` (the source fragment), `label`, `mode` (`order_request_item_mode`: `explicit | discovery`), `filters` (jsonb, `$type<OrderRequestItemFilters>` from the client-safe `src/lib/order-request.ts`), `search_phrase` (display only), `search_terms` (`text[]` — bilingual ranked-search tokens, see below), `quantity_hint` (raw qty text — no unit math), `status` (`order_request_item_status`: `pending | added | skipped`). `order_request` cascade-deletes items. All added via `npx drizzle-kit push` (additive).
- **The two modes** (the catalog is mixed-language — `name` is Latin, `category` + attribute values are Russian/UPPERCASE):
  - `explicit` — the client named a brand in Cyrillic transliteration (`Вильям лоусон`, `Хосе куэрво`). The LLM emits `search_terms` (see below); `quantity_hint`; `filters` empty.
  - `discovery` — a vague/category ask. The LLM fills `filters` from the **exact** enum values it's handed (via `listProductFilterOptions()` + `listProductCategories()`, capped at `MAX_ENUM_VALUES = 80` distinct so huge free-text-y attributes like appellation/year fall back to a phrase); price ceilings → `priceMax`. Only the bounded closed-vocabulary attributes (category/type/color/sugar/aging/bottleVolume/countryName) are exposed to the LLM; year/rating/appellation stay rep-driven. Discovery items ALSO get `search_terms` (grape/style words) to complement the structured filters.
- **Bilingual ranked search (`search_terms` + `listProducts({ terms })`)** — the catalog is mixed-language (`name` Latin, `category`/attributes Russian), client spelling is unreliable, and word order drifts, so a single transliterated phrase is brittle. Instead, per item the LLM emits `search_terms`: a deduped bag of short tokens in **both** languages — each meaningful request word kept in Russian AND its English translation/transliteration, drink-kind word always translated (Джин→Gin), name-numbers kept (135, 12), quantities/units/connector-stopwords dropped (`cleanTerms` strips `de/la/los/де/ля/…`). `listProducts` runs a **token-OR weighted-ranked** query. Per term, the best field hit is taken via `GREATEST` (so a term counts once) with **field weights** `name 3 > category/accounting 2 > attribute 1` (`TERM_MATCH_KEYS` — NOT prose `description`, to avoid noise), then scaled by a **term-distinctiveness weight** (`termWeight`: numeric `2.5` > len≥6 `1.8` > len≥4 `1.3` > short `0.8`) so rare brand words / proof-age numbers dominate common kind-words. `WHERE score > 0`, `ORDER BY score DESC, char_length(name) ASC` (shorter name = tighter-coverage tie-break). So «Джин хиога драй 135» → `[Джин,Gin,хиога,Hyoga,драй,Dry,135]` ranks **135 East Hyogo Dry Gin** at 14.9, clear of the generic «Gin Lamas … Dry» at 12.4 — the `135` (numeric) + brand-in-name win even though «хиога»≠«Hyogo» and word order differs. `search_phrase` is now display-only. **Surfacing**: while a wizard step is active and on page 1, `products/page.tsx` badges the top-ranked catalog row **"Best match"** (`showBestMatch`), so the rep's eye lands on the most probable product. **⚠️ Engine updated (Search V2, 2026-06):** the `termWeight`/`GREATEST`/`FIELD_W` weighted scorer described above is **gone** — `terms[]` now feed the same normalized + alias-expanded RRF hybrid as the manual box (see § Products "Search (Search V2)"). **The wizard is now PURE TEXT SEARCH** on the LLM's `search_terms` (cross-script normalization + brand aliases handle хиога→Hyoga, дескомб→Descombe, мюленхоф→Meulenhof deterministically) **+ price gate only.** Two deliberate constraints, both because the source catalog's structured attributes are mis-attributed and `category="Вино"` is a near-empty bucket (real wines live under Белое/Красное/Розовое): (1) `cleanFilters` **discards** the LLM's category/type/color/sugar/aging/bottleVolume/countryName — they are NOT stored, shown as chips, used as filters, or sent as hints; only `priceMin`/`priceMax` survive (a reliable client-stated ceiling, applied as a gate). (2) `cleanTerms` **drops generic drink-kind words** (вино/wine, джин/gin, виски/whisky, …) from `search_terms` — they FTS-match nearly every product of that kind and drowned out weak fuzzy brand matches (e.g. «мюленхоф Muelenhof Вино Wine» surfaced random «…Wine…» rows and buried Weingut Meulenhof; «мюленхоф Muelenhof» nails it). The rep applies a kind/colour facet by hand via the manual catalog dropdowns when wanted. `snapToVocab` + `relaxOverConstrainedFilters` were deleted (no structured filters left to snap/relax). `search_phrase` stays display-only. (`listProducts` still *accepts* `catHint`/`colorHint`/… soft-boost params + `volumeMl` for a possible future discovery surface, but the wizard sends none of them.)
- **Server** (`src/server/order-requests.ts`, `server-only`):
  - `createAndParseOrderRequest({ clientId, rawText, comment })` — inserts the request (`parsing`), runs ONE LLM split, persists the items, flips to `ready`. **Parse failure is non-fatal**: the row lands `ready` with `parse_error` set + zero items, so the rep still gets a usable wizard entry that opens the manual builder. `cleanFilters` strips the LLM's `""`/`0` sentinels (Gemini structured output can't express nullable/oneOf — same constraint as cards/deals).
  - `getOrderRequest(id)` → header + items (ordered by `ordinal`). `linkOrderToRequest(requestId, orderId)` (attach the draft + flip to `assembling`, idempotent). `updateOrderRequestItemStatus` (mark a step `added`/`skipped`). `setOrderRequestStatus` (`done`/`abandoned`/`assembling`). All org-scoped via `requireOrgContext()`.
- **API** (`src/app/api/order-requests/route.ts`, `maxDuration = 120` for the split): `POST { clientId, rawText, comment? }` → `{ id }`; `GET ?id=` → `{ request }`; `PUT { id, action }` switched on `linkOrder | itemStatus | status`. Re-exports `OrderRequestItemView` / `OrderRequestDetail`.
- **UI** — a SINGLE **"New order"** button on `/products` (gated on `!builder.isActive`) opens one dialog for both flows; the **client-request textarea is the switch**.
  - `<NewOrderDialog>` (`src/components/blocks/order-request-wizard.tsx`) — client picker (reuses `/api/orders?clientOptions=1`) + description + an OPTIONAL paste textarea. **Empty request** → manual: `onManual` opens the builder with the client + description preset (client optional, picked later if blank) — the old "New order". **Filled request** → AI: `onAssemble` POSTs to `/api/order-requests` (client required; "Analysing…" spinner during the split) and hands the request id to the assembly wizard. The submit button label follows the mode (Build order ↔ Analyse &amp; start). **Prefill props** `initialClientId` / `initialRawText` seed the form on open — used by the cards "Create order" entry point (a `new_order` card → `/products?orderFromCard=<id>` → dialog opens with the client + verbatim message prefilled; see § "Truffalo Cards" → "`new_order` cards → Create order flow").
  - **Page orchestration** (`src/app/(protected)/products/page.tsx → startAssembly`): mints an **empty draft order up front** (`POST /api/orders` with the client + comment, no items) so the order is durable + linkable from step one, links it to the request, opens the existing builder in **edit mode** on that draft (`builder.openEdit`), then walks the items. Each step's `applyItemToFilters` pre-filters the catalog table below (explicit → search box; discovery → the catalog filter selects + price), and the parsed quantity prefills `AddToOrderButton` via its new `defaultQty` prop. If the split yielded 0 items, it skips the wizard and just opens the builder with a toast.
  - `<OrderRequestWizardStrip>` — sticky strip above `OrderBuilderPanel` showing the current item (label + raw snippet + applied-filter/search chips + asked-quantity), with **Skip this / Skip the rest / Next item (→ Finish & review)**. The rep adds products from the pre-filtered catalog using the *existing* "Add to order" column; advancing stamps the item `added`/`skipped` and re-filters for the next. Finishing leaves the assembled draft in the normal builder for the standard **Save draft / Send to client** flow. The strip auto-clears when the builder closes by any path (saved/sent/discarded).
- **Reuse, not rebuild**: the per-step candidate list is the existing catalog table (ranked by the item's `search_terms` via the `?terms=` param — `applyItemToFilters` in `products/page.tsx` sets the discovery filters, the wizard item supplies the terms); adding a line is the existing `useOrderBuilder` flow; sending is the existing guest-link flow. The new code is durable rows + the parse prompt + the ranked-search + the wizard glue. **Deferred**: client resolution from venue names (rep picks the client), litre/bottle-size unit conversion (rep confirms quantity), trigram/fuzzy matching for typo'd single tokens (the bilingual token-OR already absorbs most misspellings via the other tokens).

## Truffalo Cards (LLM-generated dashboard signals)

Cards are short, structured signals the LLM emits from parsed source items — one card per item that the active rule classifies as relevant. Operators see them on the dashboard, accept or reject each, and use them as a triage feed. The Cards pipeline is the **template** the Deals discovery flow was modelled on; both share the same eligibility-filter / concurrency / per-item-stamp shape.

### Schema (`src/db/schema.ts`)

- **`card`** — one row per LLM-emitted signal. `priority: normal | high`, `category: client_activity | colleagues_activity | business_info | action_required | ambiguity | data_intelligence | momentum | log_only | new_order`. `message` is a `jsonb` column with `{ analysis, recommendation }` — kept as JSON so the shape can grow (e.g. supporting evidence links) without a migration. **`new_order` cards additionally carry `message.orderRequest`** — the VERBATIM client message from the source (Telegram `metadata_json.rawText`), stamped at generation so the card's "Create order" button can prefill the order request unchanged (`normaliseMessage` in `src/server/cards.ts` passes it through; absent on every other category). **All cards may carry `message.taskTitle`** — a concise 3-5 word action-oriented title the generation LLM summarises from the card context, used to prefill the task name on "Принять" (falls back to the category label on older cards that predate the field; `normaliseMessage` trims + passes it through). Optional `source_item_id` (FK `set null`, so card history survives if the underlying item is purged) and `rule_id` (FK `set null`, same reason). `accepted: boolean` + `rejection_reason: text` are mutually exclusive in practice — rejecting sets `rejection_reason`, accepting flips `accepted = true`. Org-scoped via `organization_id`.
- **`card_client`** — many-to-many linking a card to clients the LLM identified in the source. Composite PK `(cardId, clientId)`. Cascades on either side.
- **`card_user`** — many-to-many linking a card to users (org members) the LLM identified. **Not** the card's "creator" — cards are produced by the system pipeline, not by an org member. Composite PK `(cardId, userId)`. Cascades on either side.
- **`card_contact`** — many-to-many linking a card to the external **contact** (the source item's sender), resolved **deterministically by email** at generation time (`externalContactEmail` in `cards-generation.ts` → matched against the org's contacts-with-email roster; no LLM name-matching). Composite PK `(cardId, contactId)`, cascades on either side. Drives the "Принять" create-task prefill (the contact + its owning client). Created via `scripts/add-card-contact-table.ts` (NOT `drizzle-kit push`, which would propose dropping the Search V2 raw-DDL product columns). `CardContactRef` carries the contact's `clientId` so the dashboard can pair it with the right client for the task form's client-scoped contact picker.
- **`source_item.card_analysis_scanned_at`** — stamped by `generateCards()` after the analysis pipeline has considered the row (whether or not it produced a card). Default eligibility filter: `cardAnalysisScannedAt IS NULL OR parsedAt > cardAnalysisScannedAt`, so re-parses re-trigger analysis naturally. Cleared by `reparseSourceItem`. The "Re-analyze already-processed items" checkbox in `<ExploreSourcesDialog>` bypasses the filter for ad-hoc re-runs.

### Server (`src/server/cards.ts`)

- `listCards()` — joins to source_item + rule for context, plus the M:M client + user lists per card.
- `getCard(id)` — single-card variant for the detail page.
- `createCard()`, `updateCard()` — manual create/edit. `acceptCard(id)` flips `accepted = true`. `rejectCard(id, reason)` writes `rejection_reason` and clears `accepted`. `deleteCard(id)` hard-deletes — no soft-delete column on cards (rejection IS the soft-delete), so this is reserved for genuine cleanup.
- All mutations org-scoped via `requireOrgContext()` + `assertCardInOrg`. M:M membership updates use the same wholesale-replace pattern as `deal_contact` (delete + bulk insert) when `clientIds[]` / `userIds[]` is provided in `updateCard`.

### Cards generation (`src/server/cards-generation.ts`)

The LLM pipeline. Per-item function call, bounded concurrency, hard cap 50 per click, concurrency 3 — sized for Vercel Functions' 300s default with comfortable AI Gateway free-tier RPM headroom.

- **Eligibility** — `parseStatus = 'complete' AND r2UploadStatus = 'complete' AND markdownR2Key IS NOT NULL` plus the `card_analysis_scanned_at` filter (unless `includeAlreadyAnalyzed` is set). Optional time-window + source-id filters from the dialog.
- **Reference data passed to every LLM call** — known clients (active only) and known users (members of the org). Names join via `normaliseName` (case + whitespace normalisation) so the LLM can return human-friendly names without ID mapping. **Contacts are NOT name-matched by the LLM** — instead the org's active+initial contacts-with-email roster (`contactByEmail`) is matched **deterministically** against the source item's external sender email (`externalContactEmail`, which skips the owner's own domains via `OwnOrgIdentity`); the matched contact is linked via `card_contact`. Telegram DMs (no email party) simply produce no contact link.
- **Output schema** (`cardOutputSchema`) — Gemini-compatible flat object: `relevant: boolean`, `priority: "normal" | "high"`, `category` (UPPER_SNAKE: `CLIENT_ACTIVITY` / `COLLEAGUES_ACTIVITY` / `BUSINESS_INFO` / `ACTION_REQUIRED` / `AMBIGUITY` / `DATA_INTELLIGENCE` / `MOMENTUM` / `LOG_ONLY` / `NEW_ORDER`, plus the tolerated legacy `COLLEAGUES_UPDATE`; all normalised server-side to the snake_case enum), `message: { analysis, recommendation }`, **`taskTitle: string`** (3-5 word action title → stored in `message.taskTitle`), `clients: string[]`, `users: string[]`. For `NEW_ORDER` cards the insert path also stamps `message.orderRequest` from the source item's raw text (`metadata_json.rawText`/`text`, falling back to the markdown) — the candidate query already selects `metadataJson`. All fields REQUIRED with sentinel values for the not-relevant branch — Gemini's structured-output engine rejects nullable / oneOf union fields ("Invalid value at one_of[…]"). Server enforces via `Output.object({ schema })`; the rule prompt's `# OUTPUT CONTRACT` documents the shape so rule writers can paste it verbatim.
- **Truncation guard** — markdown clipped at 120 000 chars (~30 k tokens) to match the office parser's guard. Beyond that, even Gemini 1M-context starts to reason worse.
- **Apply path** — auto-apply (no preview-then-apply step). Each LLM call either inserts a `card` row + matching `card_client` / `card_user` edges, or counts as `skippedNotRelevant` / `skippedNoMarkdown` / `failed`. Names that don't match the org roster are **dropped silently** — the rule may surface entities the operator hasn't created yet, and we'd rather skip than 500.
- **Failure modes that DON'T stamp** the source item: R2 read failure and LLM throw. Both are transient; the next run retries them. Definitive verdicts (relevant=false, card created, empty markdown) DO stamp — single batched `UPDATE` at the end of the run, one DB round-trip regardless of batch size.
- **Result counters** (`GenerateCardsResult`): `scanned`, `cardsCreated`, `skippedNotRelevant`, `skippedNoMarkdown`, `failed`, `capped`, `errors[]` (capped at 30 entries to keep the response sane on a catastrophic batch).

### API (`src/app/api/cards/`)

- **`/api/cards`** — `GET` returns the org's cards list. `POST` creates manually. `PUT` accepts:
  - `action: "accept"` → calls `acceptCard(id)`.
  - `action: "reject"` with `rejectionReason: string` → calls `rejectCard(id, reason)`.
  - Otherwise applies a partial update via `updateCard(id, …)`.
  - `DELETE ?id=…` hard-deletes.
- **`/api/cards/generate`** — `GET` returns `{ count, cap }` for the dialog's preview chip. `POST` runs the batch with `{ from, to, sourceIds, ruleId, modelKey, includeAlreadyAnalyzed }`. `maxDuration = 300`.

### UI: Dashboard "Truffalo Cards" feed

### `support` cards → customer-support tasks

The `support` category («Поддержка», rose, **always high priority**) flags a **customer-support request** — a complaint about product quality, a damaged / wrong / missing item, a delivery problem, or a return/replacement request (distinct from `new_order`, which is the client *buying*). ## Deals (sales funnel + LLM discovery)

Sales-pipeline entity. Each deal links to one Client (required), zero-or-more Contacts (optional, multi-select), and one Funnel Stage. Tasks can carry an optional `dealId` so they show up scoped to a deal context.

### Schema (`src/db/schema.ts`)

- **`deal_funnel_stage`** — platform-wide funnel dictionary. Same `is_system = true` / `owner_organization_id = <orgId>` split as `source` / `source_template`. Today seeded with 7 system rows (Qualification 0.05 → Discovery 0.20 → Pilot 0.50 → Proposal 0.60 → Negotiations 0.80 → Closed 1.0 / Rejected 0.0) via `scripts/seed-deal-funnel-stages.ts`. `closure_probability` is `numeric(4,3)` (Drizzle returns it as a string, the API parses to number on read). `sort_order` controls UI ordering. Operators can re-point system rows to their org (set `is_system = false` + `owner_organization_id`) and re-run the seed to bootstrap fresh system rows alongside — the seed's `(name, is_system=true)` idempotency check skips existing system rows but doesn't conflict with the org-scoped clones.
- **`deal`** — name, optional description, `funnel_stage_id` (notNull, FK `restrict`), `client_id` (notNull, FK `restrict`), nullable `value` (`numeric(14,2)`) + `currency` (`text`, default `"USD"`, normalised to uppercase 3-letter ISO server-side), **`status`** (`deal_status` enum: `active | cancelled | deleted`, default `active`). Org-scoped via `organization_id`. **Lifecycle `status` replaced the old `is_cancelled` boolean** (migrated via `scripts/migrate-deal-status.ts`, which backfills `is_cancelled=true → 'cancelled'` then drops the column). It's orthogonal to the funnel `Rejected` stage (a visible sales outcome): `cancelled` = real lost/withdrawn deal (kept for analytics, surfaced via "Include cancelled"); `deleted` = test artifact / mistake (hidden from the board by default AND excluded from deal-discovery's identify/match/move logic, so a fresh re-scan can re-create it). Only `active` deals participate in discovery and the sales-funnel value. `restrict` (vs `cascade`) on the FK to client + stage matches the no-hard-delete policy — accidental deletes fail loudly instead of orphaning deals. **`reasoning` + `changes`** (both nullable text, added via `scripts/add-deal-reasoning-changes.ts`): LLM-discovery provenance — `reasoning` is the one-sentence rationale for the create/stage-move (refreshed on each move), `changes` is a short description of what the new source signal changed on a stage move (null/empty for freshly-created deals; the deal's `description` is never rewritten by a move). Null for manually-created deals. No backfill — populated only by new `generateDeals` runs. **`position`** (nullable text, added via `scripts/add-deal-position.ts` — additive, NOT `drizzle-kit push`) is a **fractional-indexing key** (rocicorp `fractional-indexing`) ordering deals *within* a funnel stage (column) for the manual kanban order; null until first dragged (board sorts null to the end by `updatedAt`). Written ONLY by `moveDeal`. Indexed `(funnel_stage_id, position)`. Backfilled once per (org, stage), most-recently-updated-first, so the board has a stable order from day one. See § "UI: Deals kanban board".
- **`deal_contact`** — composite-PK join table (mirrors `card_client` shape). Both sides cascade on hard-delete; under the no-hard-delete policy the cascade is hygiene only.
- **`task.deal_id`** — nullable FK with `onDelete: "set null"` (defensive default; not expected to fire under the no-delete policy). Cleared by `reparseSourceItem` is N/A — task rows are unrelated to source-item lifecycle.

### Funnel-stage resolution (org-or-system fallback)

`listDealFunnelStages()` in `src/server/deals.ts` does **not** union org + system rows. Instead:

1. Query active org-scoped stages for the active org. If any exist → return only those.
2. Otherwise → return system rows.

This means an org that has customised its funnel sees ONLY its own stages everywhere (form picker, kanban columns, deal-card move dropdown, discovery prompt). An org that hasn't customised inherits the seven seeded system stages. When org-scoped customisation lands later, it's purely additive — no migration of existing deals required, since each deal references a specific stage row by id regardless of bucket.

### Server (`src/server/deals.ts`)

- `listDeals({ includeCancelled?, includeDeleted? })` (default returns `active` only; each flag widens the status set — the board passes both and hides/shows client-side), `getDeal(id)`, `createDeal()` (always `status: 'active'`), `updateDeal()` (accepts `status?: DealStatus`, validated against the enum), `setDealStatus(id, status)` (the quick cancel/delete/restore shortcut, mirrors task's `updateTaskStatus`).
- `listDealClientOptions()` / `listDealContactOptions(clientId?)` — picker scope is `active` + `initial`, excludes `suspended`. The client picker explicitly does NOT filter by `status = "active"` because operators routinely want to attach deals to auto-discovered clients before flipping them active.
- `listDealFunnelStages()` — see above.
- `moveDeal(id, { funnelStageId, position })` — kanban-drag persistence: validates tenant + stage, writes `funnel_stage_id` + the **client-computed** fractional-index `position`. The ONLY writer of `deal.position`. `updatedAt` bumps via `$onUpdate` (a drag counts as a touch, same as the stage-move dropdown). `listDeals` / `getDeal` now surface `position` on `DealRow`.
- All mutations org-scoped via `requireOrgContext()` + `assertDealInOrg` / `assertClientInOrg` / `assertFunnelStageAccessible` / `assertContactsInOrg`. Currency normalised to 3-letter ISO (throws on bad input); value normalised to fixed 2-decimal string (drizzle's `numeric` accepts string or number — fixed format keeps the column consistent).
- `updateDeal` replaces deal-contact membership wholesale via delete + bulk insert when `contactIds` is provided. Cardinality is small; diffing was overkill.

### API (`src/app/api/deals/route.ts`)

- `GET /api/deals` — list. Query knobs: `?includeCancelled=1`, `?includeDeleted=1`, `?clientOptions=1`, `?contactOptions=1&clientId=…`, `?funnelStages=1`, `?id=…` (single deal).
- `POST /api/deals` — create. Required: `name`, `clientId`, `funnelStageId`. Optional: `description`, `contactIds[]`, `value`, `currency`.
- `PUT /api/deals` — update. Two shortcut branches: `moveOnly: true` + `funnelStageId` + `position` (the **kanban-drag** path → `moveDeal`, so a drag never touches name/value/contacts) and `statusOnly: true` + `status` (quick cancel/delete/restore, mirrors tasks' `statusOnly`). Otherwise applies any subset of `{ name, description, funnelStageId, clientId, contactIds, value, currency, status }` to the row. `status` is validated against the `deal_status` enum on all paths.

### Sales funnel value calculation

```
salesFunnelValue = Σ (deal.value × deal.funnelStageProbability) for all non-cancelled deals
```

- Per-deal probability comes from the joined `funnelStageProbability` field on `DealRow` (so deals on stages outside the resolved bucket still contribute correctly).
- Deals with `value === null` are skipped (not coerced to 0). Cancelled deals are excluded.
- Filters and stage-tab selection do NOT change this number — it's a board summary, not a filtered total. Refreshes via `refreshAll` whenever a deal is created, edited, moved, or cancelled.

### Deal discovery from sources (LLM)

Manual button on the Deals tab ("Discover from sources", left of "+ New deal"). Mirrors the Cards-from-sources pattern (`src/components/blocks/explore-sources-dialog.tsx` + `src/server/cards-generation.ts`) and is implemented in:

- **Server**: `src/server/deals-discovery.ts` (`previewDealDiscoveryCandidates`, `generateDeals`).
- **Route**: `POST/GET /api/deals/discover`. `maxDuration = 300`.
- **Client**: `src/components/blocks/discover-deals-dialog.tsx`.

**Eligibility filter** (same shape as cards-generation):
- `parseStatus = 'complete' AND r2UploadStatus = 'complete' AND markdownR2Key IS NOT NULL`.
- By default also `dealAnalysisScannedAt IS NULL OR parsedAt > dealAnalysisScannedAt` — bypassed when the dialog's "Re-analyze already-processed items" checkbox is ticked.
- Hard cap of 50 items per click, concurrency 3.

**Dry-run mode (`dryRun` on `GenerateDealsInput`)** — the rule-testing path. Runs the full LLM pass and computes every decision (counters + a `plannedActions: PlannedDealAction[]` list of would-be CREATE/MOVE actions) but writes **nothing**: no deal/contact inserts, no stage updates, and crucially **no `dealAnalysisScannedAt` stamp**, so the same items stay eligible for the next pass. The dialog renders the planned actions inline and keeps itself open (doesn't refresh the board). Iterate on rule wording with dry-run → untick → run for real. The result carries `dryRun: boolean` so the client branches on it.

**Create-side dedup** — every CREATE is checked against a `${clientId}::${normaliseName(name)}` key set seeded from all **non-deleted** deals in the org (active + cancelled) and grown in-memory as the run creates deals (the key is added before the insert `await` to close the check-then-act race between concurrent workers). A collision is counted as `skippedDuplicate`, logged, and stamped. `deleted` deals are deliberately excluded from the key set, so after marking a test run's deals `deleted` a fresh re-scan re-creates them cleanly.

**`source_item.deal_analysis_scanned_at`** column tracks discovery state (parallels `card_analysis_scanned_at`). Cleared by `reparseSourceItem` so a manual re-parse re-triggers discovery.

**Bootstrap cutoff** — `scripts/backfill-deal-discovery-cutoff.ts` stamps `deal_analysis_scanned_at = NOW()` on every row whose `source_created_at < 2026-05-05` so the first runs don't try to chew through the entire historical archive. Ran once during initial deployment (~682 rows). Re-runnable safely (skips already-stamped rows). Raising the cutoff requires editing the script's `DEAL_DISCOVERY_BOOTSTRAP_CUTOFF` constant + a re-run.

**Reference data passed to every LLM call**:
- Known clients (active + initial) — names only.
- Known contacts (active + initial) — names + linked client name.
- Open deals — `status = 'active'`, non-terminal (`0 < closureProbability < 1`). Cancelled + deleted are out of the match set entirely. Probability bounds make the filter robust to org-customised funnel names that don't include literal "Closed" / "Rejected". Each open deal renders as a multi-line block: `name`, `current stage` (with closure probability %), `client`, optional `description` (200-char clamp). Multi-line so the rule prompt can refer to fields by their explicit labels.
- Funnel stages — resolved via the org-or-system fallback above; passed with their closure-probability percentages so the model can reason about move direction.

**LLM output schema** (enforced server-side via `Output.object({ schema })`, mirrored in the prompt's `# OUTPUT CONTRACT`). Single flat object with sentinel values for the branches the chosen `action` doesn't touch — Gemini's structured output can't handle nullable / oneOf union fields. Server discards branch-irrelevant fields based on `action`:

```
{
  "relevant": boolean,
  "action": "CREATE" | "UPDATE_STAGE" | "SKIP",

  // CREATE branch
  "newDealName":         string,
  "newDealDescription":  string,
  "newDealClientName":   string,    // must exactly match a Known Client
  "newDealStageName":    string,    // must match a Funnel Stage
  "newDealValue":        number,    // 0 if unknown
  "newDealCurrency":     string,    // 3-letter ISO, "" if unknown
  "newDealContactNames": string[],  // must match Known Contact names

  // UPDATE_STAGE branch
  "matchedDealName":     string,    // must match an Open Deal name
  "updatedStageName":    string,    // must match a Funnel Stage (the NEW stage)
  "changes":             string,    // UPDATE_STAGE only: what the new signal changed (logged on the deal; description NOT rewritten). "" for CREATE/SKIP

  "reasoning": string               // one sentence on why created / stage moved
}
```

**Apply path** — the server runs the action immediately (mirrors Cards' auto-apply, not Discover-Clients' preview-then-apply):

- `CREATE` → resolve client + stage + contacts via case-insensitive normalised name lookup (`normaliseName`). Unknown client / unknown stage → counted in the corresponding `skippedUnknown*` counter, logged in `errors[]`, source item still stamped (won't retry by default; re-parse or `includeAlreadyAnalyzed` to re-run). A `${clientId}::${normalisedName}` collision with an existing non-deleted deal → `skippedDuplicate` (logged + stamped, no insert). Currency falls through to `"EUR"` if blank or invalid; value rounded to 2 decimals. Inserts into `deal` (`status: 'active'`) + `deal_contact`.
- `UPDATE_STAGE` → resolve matched deal name + new stage via the same normalised lookup. Unknown matched deal / unknown new stage → counted + logged + stamped. Successful match runs an UPDATE on `deal.funnel_stage_id` plus refreshes `deal.reasoning` + `deal.changes` from the model output — the deal's `description` is deliberately left untouched.
- `SKIP` (or `relevant: false`) → counted, stamped.

**Failure modes that DON'T stamp** the source item: R2 read failure and LLM throw. Both are transient; the next run retries them. Same pattern as cards-generation.

**Result counters** (`GenerateDealsResult`): `scanned`, `dealsCreated`, `stageUpdates`, `skippedNotRelevant`, `skippedNoMarkdown`, `skippedUnknownClient`, `skippedUnknownStage`, `skippedUnknownDeal`, `skippedDuplicate`, `failed`, `capped`, `dryRun`, `plannedActions[]` (dry-run only), `errors[]` (capped at 30 entries to keep the response sane on a catastrophic batch). On a dry run the `dealsCreated` / `stageUpdates` counters reflect would-be actions, not committed ones.

**Rule writing** — operators write a Custom rule on `/rules` with the body describing when to CREATE vs UPDATE_STAGE vs SKIP. The reference prompt at `refs/deals-rule.md` is the canonical example. Key invariants to bake into the rule:
- Stage moves must be **forward-only** (toward higher closure probability) unless moving to Rejected. The move-direction guard isn't enforced server-side — it's a rule-prompt concern.
- If the source describes a sales-relevant signal but no Known Client matches the company, the rule should set `action: SKIP` rather than emit a CREATE the server will then drop.
- If the source mentions an existing open deal but contains no signal that would justify any of the stage moves, the rule should set `action: SKIP` (no-op UPDATE_STAGE is wasteful).
- New deals should default `newDealStageName` to the first funnel stage of the resolved funnel (Qualification in the seeded funnel).
